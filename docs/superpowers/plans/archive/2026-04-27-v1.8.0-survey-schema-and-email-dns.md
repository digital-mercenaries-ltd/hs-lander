# Plan: v1.8.0 — Survey schema, email DNS preflight, capture redirect default, survey-submit.js, survey_completed flag, phantom-form resolution

**Date:** 2026-04-27
**Status:** Complete (shipped as v1.8.0 on 2026-04-27)
**Scope:** Framework. Module input schema extensions (`survey_fields` field types, `custom_properties` enum), new scaffold JS, capture form default behaviour change, new auto-added boolean property, new preflight emit (`PREFLIGHT_EMAIL_DNS`), reference-doc updates.
**Target release:** v1.8.0 (minor — module input contract changes + new preflight emit + capture default behaviour change)
**Dependency:** v1.7.1 shipped

## Problem

A downstream Starter deploy reaching live state surfaced six gaps that compound to "the survey funnel is non-functional and email deliverability is silently broken":

### Issue 1 — Survey form fields can only be `single_line_text`

`terraform/modules/landing-page/forms.tf` schema for `survey_fields`:

```hcl
variable "survey_fields" {
  type = list(object({
    name     = string
    label    = string
    type     = string   # ← currently only "single_line_text" works
    required = bool
  }))
}
```

Real surveys need dropdowns (single-choice from list), checkboxes (multi-choice), and radios (single-choice with all options visible). Free-text answers are usable for "name", "company", but not for "what's your accent?" or "what tools do you currently use?" — those become typo-laden, case-mismatched, unconstrained free-text strings that destroy reporting and segmentation.

### Issue 2 — `custom_properties` are plain strings even when forms have constrained values

The companion CRM contact properties (defined per project in `custom_properties`) are also `type: string, fieldType: text`. So even if survey form fields became dropdowns submitting clean values like `"American"` / `"British"`, the values land in CRM properties typed as free-text. HubSpot reporting can't enumerate them; segmentation by exact match is fragile (case sensitivity, whitespace, etc.).

### Issue 3 — The "survey form" is a phantom

Terraform creates `restapi_object.survey_form` (HubSpot form id `<...>`), but the scaffold thank-you page renders a static HTML form with `<select>`, `<input type="checkbox">`, `<input type="radio">` — handcrafted, no `hbspt.forms.create` embedding the Terraform-defined form. The static form has no `action` attribute, no JS submit handler. Submitting it does nothing — the data is silently discarded. Users see "Send answers" → page navigation; receivers (us) see nothing.

This was the bigger of two bugs surfaced by a downstream deploy. The HubSpot-defined form exists in the portal but is never embedded; the rendered form looks right but submits to the void.

### Issue 4 — No email handoff between capture and survey

The capture form's `postSubmitAction` is currently:

```hcl
postSubmitAction = {
  type  = "thank_you"
  value = "Thanks for submitting the form."
}
```

This shows HubSpot's inline thank-you snippet on the same page. There's no automatic redirect to `/thank-you`, and no way to carry the user's email to the survey form for pre-population or attribution. Users only reach the thank-you page via the welcome email's CTA, hours later — and at that point we have no identifier to attach their survey responses to.

### Issue 5 — Survey completion is invisible

No CRM property records whether a contact completed the survey. Segmentation queries like "send second nudge to contacts who *didn't* complete the survey" can't be expressed without one. Should be a boolean property like `<slug>_survey_completed` flipped on submit.

### Issue 6 — Email deliverability is silently broken on misconfigured DNS

A downstream Starter deploy went live with a broken SPF record (`~all` mid-record before HubSpot's `include:`, so the HubSpot include was never evaluated). HubSpot mail wasn't authorised to send as the project's domain — Gmail/Apple Mail would tag with "via amazonses.com" and weight reputation accordingly. DKIM was fine but the initial probe missed it because `dig +short ANY <selector>._domainkey.<domain>` returns empty under RFC 8482 (Cloudflare refuses ANY queries).

`scripts/preflight.sh` checks the project domain's CNAME (landing page hosting) but does **not** check email-auth records (SPF / DKIM / DMARC) for the `email_reply_to` domain. Email deliverability silently degrades for weeks until users notice nothing's arriving.

### Bonus — tier-classifier informed-guess values can now be partly verified

The v1.7.0 plan shipped `scripts/lib/tier-classify.sh` with informed-guess `accountType` values pending real-portal verification. The v1.7.0 deploy round used a real Starter portal — the Starter row of the table can now be confirmed. Pro and Enterprise rows still need access to those tiers.

## Empirical prerequisites

Two probes; both blockers.

### Prerequisite A — Capture-form merge-token syntax for `redirect_url`

`postSubmitAction.type = "redirect_url"` accepts a `value` URL with merge tokens that HubSpot substitutes from form submission. The exact token syntax for "the email field's submitted value" is one of: `{{email}}`, `{{form_field.email}}`, `{{contact.email}}`, or possibly `{email}`. Probe before implementing:

```bash
# Create a test form with a redirect_url postSubmitAction using each candidate
# token, submit a real test entry, observe the resulting URL.
# Or check HubSpot Forms v3 API documentation under "redirect URL personalization".
```

If the probe reveals the syntax differs from `{{email}}`, update Fix 4 below accordingly. Capture the verified syntax in `references/hubspot-api-quirks.md`.

### Prerequisite B — `dig` DKIM probe that actually works for HubSpot's selectors

Empirical evidence from a recent deploy round:
- HubSpot uses portal-id-suffixed DKIM selectors: `hs1-<portal-id>._domainkey.<domain>` and `hs2-<portal-id>._domainkey.<domain>` (CNAME records).
- `dig +short ANY <selector>._domainkey.<domain>` returns empty when the authoritative server refuses ANY (RFC 8482; Cloudflare default).
- `dig +short CNAME <selector>._domainkey.<domain>` works correctly.

Probe at least two HubSpot portals to confirm the `hs1-<portal>` / `hs2-<portal>` pattern is portal-agnostic (one selector pattern across portals, varying only by portal ID). If different selector schemas exist across portals, the preflight check needs to probe a list of candidate selectors. The two known portals to date both use `hs1-<portal-id>` / `hs2-<portal-id>`.

## Design

### Fix 1 — Extend `survey_fields` schema with field types, options, and "Other" overflow

`terraform/modules/landing-page/variables.tf`:

```hcl
variable "survey_fields" {
  description = <<EOT
Survey form fields. Each field declares:
- name: HubSpot internal field name (matches a custom_properties entry by name)
- label: visible label
- type: one of "single_line_text", "dropdown", "multiple_checkboxes", "radio"
- required: bool
- options: list of strings — REQUIRED when type ∈ {dropdown, multiple_checkboxes, radio}; ignored otherwise
- other_overflow: bool (default false) — when true AND options[] contains "Other"
    (or "Other (please specify)" or any case-insensitive variant), the rendered form
    shows a paired text input revealed when "Other" is selected, capturing into a
    sibling CRM property named "<field.name>_other" (auto-created as a string property;
    skill puts it in custom_properties). Ignored when type = "single_line_text".
EOT
  type = list(object({
    name           = string
    label          = string
    type           = string
    required       = bool
    options        = optional(list(string), [])
    other_overflow = optional(bool, false)
  }))
  default = []
}
```

`forms.tf` rendering: when `other_overflow = true`, the Terraform-defined HubSpot survey form gains a sibling `<field.name>_other` field (single_line_text, hidden) so Forms Submissions API submissions with the overflow value validate. The static form's overflow markup and the JS reveal logic are skill / framework-scaffold concerns (see Fix 3 + skill v1.8.0 plan §Step 7).

`forms.tf`'s field rendering switches on `field.type`:

| `field.type` value | HubSpot field shape |
|---|---|
| `single_line_text` | current behaviour — text input, no options |
| `dropdown` | `fieldType = "single_line_text"`, `options = [{label, value, displayOrder, doubleData}, ...]` per HubSpot Forms v3 dropdown shape |
| `multiple_checkboxes` | `fieldType = "single_line_text"` (legacy quirk; HubSpot stores multi-select as `;`-joined), `options` populated, `field.type = "multi_line_text"` if HubSpot needs that for multi-select rendering — verify against probe |
| `radio` | `fieldType = "single_line_text"`, `options` populated, plus appropriate display-mode flag |

**Prerequisite for implementer:** probe HubSpot Forms v3 API for the canonical create-payload shape for each of `dropdown`, `multiple_checkboxes`, `radio` field types. Capture the exact field-shape requirements (e.g. some require `formField` parent vs `fieldType` direct; some require `options[].value` vs `options[].displayOrder`). Paste the verified shapes into the `forms.tf` rendering logic. The HubSpot UI for "edit form fields" is the source of truth — create a sample form with one of each field type, GET it via API, compare structures.

Validation: the framework rejects `type ∈ {dropdown, multiple_checkboxes, radio}` with empty `options`. Terraform `validation` blocks on the variable enforce this at plan time, surfacing errors before apply.

### Fix 2 — Extend `custom_properties` schema with enumeration support

`variables.tf`:

```hcl
variable "custom_properties" {
  description = <<EOT
CRM contact properties created idempotently by the account-setup module.
- name: HubSpot internal property name
- label: visible label
- type: "string" | "enumeration" | "bool" | "number"
- fieldType: "text" | "select" | "checkbox" | "radio" (auto-defaulted from type if omitted)
- options: list of strings — REQUIRED when type = "enumeration"; ignored otherwise
EOT
  type = list(object({
    name      = string
    label     = string
    type      = string
    fieldType = optional(string, "text")
    options   = optional(list(string), [])
  }))
  default = []
}
```

`account-setup` module's property-creation HCL switches on `type`:
- `"string"`: current behaviour.
- `"enumeration"`: `type = "enumeration"`, `fieldType = "select" | "radio" | "checkbox"` per `fieldType` input, `options = [{label, value, displayOrder, hidden}, ...]` per HubSpot CRM v3 enumeration property shape.
- `"bool"`: `type = "bool"`, `fieldType = "booleancheckbox"`.
- `"number"`: `type = "number"`, `fieldType = "number"`.

Survey form fields with `type ∈ {dropdown, multiple_checkboxes, radio}` should map to `custom_properties` entries with `type = "enumeration"` and matching `options` lists. The skill (separate plan) generates these in lockstep so form fields and CRM properties have identical option sets.

When a survey field has `other_overflow = true`, the skill generates an **additional** `custom_properties` entry for the overflow value: `name = "<field.name>_other"`, `type = "string"`, `fieldType = "text"`, `options = []`. The string-typed sibling captures the free-text typed when the user selects "Other" from the enumeration; reporting on this property is unconstrained (free-text), accepted as the cost of capturing the overflow at all.

Implementer probes HubSpot CRM v3 properties API for the canonical enumeration property shape (similarly to Fix 1).

### Fix 3 — `scaffold/src/js/survey-submit.js` (new file) + thank-you template wiring

New file `scaffold/src/js/survey-submit.js`:

```javascript
// survey-submit.js — wires the static survey form on the thank-you page to
// HubSpot's Forms Submissions API, carrying the email captured upstream via
// URL parameter (?email=...).
//
// Generated by hs-lander scaffold. Token __PORTAL_ID__ and __SURVEY_FORM_ID__
// substituted at build time.

(function() {
  const PORTAL_ID = "__PORTAL_ID__";
  const SURVEY_FORM_ID = "__SURVEY_FORM_ID__";
  const SURVEY_COMPLETED_PROPERTY = "__PROJECT_SLUG___survey_completed";

  function getEmailFromURL() {
    const params = new URLSearchParams(window.location.search);
    return params.get("email") || "";
  }

  function collectFields(form) {
    const fields = [];
    const seen = new Set();

    // Single-value inputs (select, radio, single text)
    form.querySelectorAll("select, input[type='radio']:checked, input[type='text']").forEach(el => {
      if (!el.name || seen.has(el.name)) return;
      // Skip "Other" overflow text inputs at this stage; collect them after
      if (el.dataset && el.dataset.otherOverflowFor) return;
      fields.push({ name: el.name, value: el.value });
      seen.add(el.name);
    });

    // Multi-checkbox values — semicolon-joined per HubSpot multi-select convention
    const checkboxNames = new Set();
    form.querySelectorAll("input[type='checkbox']:checked").forEach(el => {
      if (el.name) checkboxNames.add(el.name);
    });
    checkboxNames.forEach(name => {
      const values = Array.from(form.querySelectorAll(`input[type='checkbox'][name='${name}']:checked`))
        .map(el => el.value);
      fields.push({ name, value: values.join(";") });
    });

    // "Other" overflow text inputs — paired with a primary field; included only when
    // the primary field's selected value is "Other" (case-insensitive)
    form.querySelectorAll("[data-other-overflow-for]").forEach(overflowEl => {
      const primaryName = overflowEl.dataset.otherOverflowFor;
      const primary = fields.find(f => f.name === primaryName);
      if (primary && /^other(\s|\(|$)/i.test(primary.value) && overflowEl.value.trim()) {
        fields.push({ name: `${primaryName}_other`, value: overflowEl.value.trim() });
      }
    });

    return fields;
  }

  // Reveal/hide "Other" overflow inputs when the paired primary field's value changes
  function bindOverflowReveal(form) {
    form.querySelectorAll("[data-other-overflow-for]").forEach(overflowEl => {
      const primaryName = overflowEl.dataset.otherOverflowFor;
      const primaryEls = form.querySelectorAll(`[name='${primaryName}']`);
      const update = () => {
        const primaryValue = (function() {
          const select = form.querySelector(`select[name='${primaryName}']`);
          if (select) return select.value;
          const radio = form.querySelector(`input[type='radio'][name='${primaryName}']:checked`);
          return radio ? radio.value : "";
        })();
        overflowEl.hidden = !/^other(\s|\(|$)/i.test(primaryValue);
      };
      primaryEls.forEach(el => el.addEventListener("change", update));
      update();
    });
  }

  async function submitSurvey(form, event) {
    event.preventDefault();

    const email = getEmailFromURL();
    if (!email) {
      console.error("survey-submit: no ?email= URL parameter; cannot attribute submission");
      // TODO: surface an inline error UI rather than just console
      return;
    }

    const fields = [
      { name: "email", value: email },
      ...collectFields(form),
      { name: SURVEY_COMPLETED_PROPERTY, value: "true" }
    ];

    const payload = {
      submittedAt: Date.now(),
      fields,
      context: {
        pageUri: location.href,
        pageName: document.title
      }
    };

    try {
      const response = await fetch(
        `https://api.hsforms.com/submissions/v3/integration/secure/submit/${PORTAL_ID}/${SURVEY_FORM_ID}`,
        {
          method: "POST",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify(payload)
        }
      );
      if (!response.ok) throw new Error(`HTTP ${response.status}`);

      // Switch the panel to its is-submitted state (existing scaffold CSS handles styling).
      form.classList.add("is-submitted");
    } catch (err) {
      console.error("survey-submit: failed", err);
      // TODO: inline error UI
    }
  }

  document.addEventListener("DOMContentLoaded", () => {
    const form = document.querySelector("[data-survey-form]");
    if (!form) return;
    bindOverflowReveal(form);
    form.addEventListener("submit", (e) => submitSurvey(form, e));
  });
})();
```

Tokens: `__PORTAL_ID__`, `__SURVEY_FORM_ID__`, `__PROJECT_SLUG__` substituted by `build.sh` (existing pipeline; check the substitution patterns include these — add to sed rules if missing).

`scaffold/src/templates/thank-you.html` updates:
- Survey `<form>` element gains `data-survey-form` attribute
- `<script src="{{ get_asset_url('__DM_PATH__/js/survey-submit.js') }}" defer></script>` added before `{{ standard_footer_includes }}`
- Submit button kept; the JS preventDefault handles the actual submission

The "secure submit" endpoint requires no authentication (it's the same endpoint HubSpot's own embedded forms use under the hood); the form ID acts as the authorisation token.

Reference doc `references/forms-submissions-api.md` (new) documents the pattern: when to use Submissions API vs. when to embed via `hbspt.forms.create`. Embedded forms are easier (HubSpot handles validation, consent UI, etc.) but constrained to HubSpot's rendering. Submissions API gives the project full control over rendered form HTML at the cost of writing the submit-handler JS. The framework defaults to embedded for the capture form (rendered by HubSpot) and Submissions API for the survey form (rendered by the project's thank-you template, which has design opinions).

### Fix 6 — Resolve the phantom HubSpot survey form

Two viable paths:

**Path A — Embed via `hbspt.forms.create` instead of static markup.** Replace the scaffold's static `<form>` on `thank-you.html` with a `<div id="survey-form-embed"></div>` plus a `hbspt.forms.create({...})` call targeting that div. This makes the HubSpot-defined survey form the only source of truth: extending field types becomes a Forms-v3 schema change in `forms.tf`, and rendering follows automatically. Pros: zero JS to maintain. Cons: surrenders rendering control to HubSpot's CSS — the scaffold's branded thank-you-page design would need to recover via CSS overrides.

**Path B — Keep static markup; document the survey form as "API submission target only".** The HubSpot-defined form stays (Forms Submissions API requires a form ID) but its purpose is documented as: "exists to give submissions a form_id; never rendered as embed." Pros: keeps full design control. Cons: framework now has two contracts — visible static markup (scaffold) and invisible HubSpot form (Terraform). They must stay schema-aligned (field names match property names) or submissions silently fail validation.

**Recommendation: Path B.** The downstream deploy proved that branded thank-you-page design matters; embedded-form CSS recovery is a real cost. Path B's two-contract risk is mitigated by:
- Skill (separate plan) generates static form and Terraform survey form together with matched field schemas.
- v1.7.0+ `test-deployment.sh` can add a check that the static form's `<input>` `name` attributes match the HubSpot form's field names via API (catches schema drift).

### Fix 7 — `PREFLIGHT_EMAIL_DNS` (new preflight emit)

Extend `scripts/preflight.sh` with email-auth DNS checks for the `email_reply_to` domain.

```bash
# Extract domain from email_reply_to.
EMAIL_DOMAIN="${EMAIL_REPLY_TO##*@}"

# SPF
SPF_RECORD=$(dig +short TXT "$EMAIL_DOMAIN" | grep -i 'v=spf1' | head -1)
if [[ -z "$SPF_RECORD" ]]; then
  PREFLIGHT_EMAIL_DNS="spf-missing"
elif ! grep -q "include:${HUBSPOT_PORTAL_ID}.spf04.hubspotemail.net" <<< "$SPF_RECORD"; then
  PREFLIGHT_EMAIL_DNS="spf-no-hubspot-include"
elif [[ "$SPF_RECORD" =~ (\~|\?|\+|-)all\ +include: ]]; then
  PREFLIGHT_EMAIL_DNS="spf-all-mid-record"
fi

# DKIM — typed CNAME query, never ANY (RFC 8482 refusal)
DKIM_HS1=$(dig +short CNAME "hs1-${HUBSPOT_PORTAL_ID}._domainkey.${EMAIL_DOMAIN}")
DKIM_HS2=$(dig +short CNAME "hs2-${HUBSPOT_PORTAL_ID}._domainkey.${EMAIL_DOMAIN}")
if [[ -z "$DKIM_HS1" || -z "$DKIM_HS2" ]]; then
  PREFLIGHT_EMAIL_DNS_DKIM="dkim-missing"
fi

# DMARC (warn-only)
DMARC=$(dig +short TXT "_dmarc.${EMAIL_DOMAIN}" | grep -i 'v=DMARC1' | head -1)
if [[ -z "$DMARC" ]]; then
  PREFLIGHT_EMAIL_DNS_DMARC="dmarc-missing"
fi
```

Emit values:
- `PREFLIGHT_EMAIL_DNS=ok` — all three present and well-formed
- `PREFLIGHT_EMAIL_DNS=spf-missing` — no SPF record at apex
- `PREFLIGHT_EMAIL_DNS=spf-no-hubspot-include` — SPF exists but doesn't include the portal-specific `<id>.spf04.hubspotemail.net`
- `PREFLIGHT_EMAIL_DNS=spf-all-mid-record` — SPF exists with HubSpot include but `all` token is not last
- `PREFLIGHT_EMAIL_DNS_DKIM=dkim-missing` — one or both DKIM CNAME selectors absent
- `PREFLIGHT_EMAIL_DNS_DMARC=dmarc-missing` — no DMARC record (warn only — preflight passes)

Skill consumes (separate plan) and coaches the fix. Reference doc `references/email-auth-dns.md` (new) documents the SPF / DKIM / DMARC requirements with fix examples.

**Region-specific HubSpot SPF includes** — `references/email-auth-dns.md` (new) contains a concrete table the preflight script consults via `HUBSPOT_REGION` from project config:

| `HUBSPOT_REGION` | Required SPF include | Source / verification |
|---|---|---|
| `eu1` | `<portal-id>.spf04.hubspotemail.net` | Verified against EU1 portal during recent deploy round |
| `na1` | `<portal-id>.spfXX.hubspotemail.net` (XX TBD) | Probe via HubSpot's "Email sending domains" UI on a NA1 portal — capture the include the UI displays in its DNS instructions |
| `apac` (when introduced) | TBD | Probe at first APAC consumer |

`preflight.sh` reads `HUBSPOT_REGION` from project config and substitutes the matching include into the SPF check. If `HUBSPOT_REGION` is unset or unknown, preflight emits `PREFLIGHT_EMAIL_DNS=region-unknown` and the skill coaches the consumer to set it; check skipped for unknown regions rather than producing misleading "spf-no-hubspot-include" errors.

Implementer probe step (blocker before shipping the NA1 row): verify the actual NA1 SPF include hostname by running through HubSpot UI's "Connect a sending domain" flow on a NA1 portal and recording the DNS instruction it provides. The above table's NA1 row is a placeholder until that probe completes; v1.8.0 ships with at least the EU1 row verified.

### Fix 8 — Schema-alignment test in `tests/test-deployment.sh`

New test that asserts the static survey form's `<input name="...">` attributes match the HubSpot survey form's field names (via API) and the `custom_properties` names (via the same API). Catches the failure mode "static form looks fine but submits to nowhere because field names drifted out of alignment between the three layers" — the worst kind of silent regression because every individual layer passes its own checks.

Pseudocode added to `tests/test-deployment.sh`:

```bash
# Schema alignment: static thank-you survey form vs HubSpot survey form vs custom_properties
test_schema_alignment() {
  if [[ "${INCLUDE_SURVEY:-false}" != "true" ]]; then
    echo "  Skip: schema alignment (include_survey=false)"
    return 0
  fi

  local survey_form_id; survey_form_id=$(jq -r '.survey_form_id.value' "$TF_OUTPUT_JSON")

  # Names rendered in the static form
  local static_names; static_names=$(grep -oE 'name="[a-zA-Z0-9_]+"' "$LANDING_HTML_FILE" "$THANKYOU_HTML_FILE" \
    | sed 's/.*name="\([^"]*\)"/\1/' | sort -u | grep -E "^${PROJECT_SLUG}_")

  # Names declared on HubSpot survey form (excluding email gateway and project_source segmentation)
  local hubspot_names; hubspot_names=$(bash "$FRAMEWORK_HOME/scripts/hs-curl.sh" GET "/marketing/v3/forms/$survey_form_id" \
    | jq -r '.fieldGroups[].fields[].name' | sort -u \
    | grep -E "^${PROJECT_SLUG}_")

  # Names declared as custom_properties (excluding the auto-added _survey_completed)
  local prop_names; prop_names=$(bash "$FRAMEWORK_HOME/scripts/hs-curl.sh" GET "/crm/v3/properties/contacts" \
    | jq -r --arg slug "$PROJECT_SLUG" '.results[] | select(.name | startswith($slug + "_")) | select(.name != $slug + "_survey_completed") | .name' \
    | sort -u)

  # Three-way diff: every name in any one layer should appear in all three
  local missing
  missing=$(comm -3 <(echo "$static_names") <(echo "$hubspot_names"))
  if [[ -n "$missing" ]]; then
    echo "  FAIL: schema alignment — static form names ↔ HubSpot form names mismatch:"
    echo "$missing" | sed 's/^/    /'
    return 1
  fi

  missing=$(comm -3 <(echo "$static_names") <(echo "$prop_names"))
  if [[ -n "$missing" ]]; then
    echo "  FAIL: schema alignment — static form names ↔ custom_properties mismatch:"
    echo "$missing" | sed 's/^/    /'
    return 1
  fi

  echo "  OK: schema alignment ($(echo "$static_names" | wc -l | tr -d ' ') survey fields aligned across static/HubSpot/CRM)"
}
```

Test gate accommodates "Other" overflow: when `static_names` includes `<field>_other`, `hubspot_names` and `prop_names` should also include it (per Fix 1 + Fix 2 sibling-property generation).

This is the regression-prevention check the v1.7.0 plan deferred to v1.9.0; promoting to v1.8.0 because the v1.8.0 changes themselves create new alignment requirements (the static form is no longer ornamental; it's the real submission path), and shipping the schema work without the alignment check would re-introduce the same silent-failure mode this plan is trying to eliminate.
