# Plan: v1.4.0 — API Drift Fixes

**Date:** 2026-04-22
**Status:** Pending
**Scope:** Framework. Four coordinated fixes to the landing-page module and account-setup module to close HubSpot API drift surfaced by the first real deploy (Heard, 2026-04-22) plus follow-up probing (same date).
**Target release:** v1.4.0 — **breaking change** per semver (landing-page module gains a required variable; email payload structure changes materially).

## Context

First apply against a live HubSpot portal (147959629, dml account) revealed three failures and an initial hypothesis of one or two more. Direct API probing of the partially-created resources (2026-04-22) confirmed four distinct drift issues:

1. **Forms API v3 requires `formType`.** Confirmed by apply failure; fix is trivial.
2. **List creation races `project_source` property.** Confirmed by apply failure; fix requires inter-module dependency wiring.
3. **Marketing email content and sender discarded.** Confirmed by probing the created email (ID 395263518907): `content.html` field silently ignored, `fromName` and `replyTo` stored as `null`, `type = "REGULAR"` coerced to `"BATCH_EMAIL"`. The resource exists but is a hollow shell — empty body, no sender.
4. **Landing page `slug = ""` does not yield root-of-domain.** Confirmed by probing the created page (ID 395236739258): HubSpot replaced the empty slug with `"-temporary-slug-<uuid>"`. Page is live but at an un-shareable URL.

This plan addresses all four in one release. The combined scope is genuinely breaking:

- New required variable on `landing-page` module (`project_source_property_id`) — every scaffolded project needs to wire it in
- `emails.tf` payload restructured to match modern HubSpot API
- Empty-slug semantics for landing pages changes

Ship as v1.4.0, not v1.3.1.

## Fix 1 — Forms: add `formType`

**File:** `terraform/modules/landing-page/forms.tf`

Both `restapi_object` resources (`capture_form` and `survey_form`) gain `formType = "hubspot"` inside the `data = jsonencode({ ... })` block, alongside `name`, `createdAt`, `fieldGroups` etc.

Probe evidence: TSC's existing capture form (`420317fa-b7c9-4f30-802c-14ecd6521c1a`) has `formType = "hubspot"`. Same value works on the current API. No further investigation needed.

## Fix 3 — Marketing email: restructure payload

**File:** `terraform/modules/landing-page/emails.tf`

### Evidence from live probes

Heard's failing email (`395263518907`) created as:
- `type: "BATCH_EMAIL"` (framework sent `"REGULAR"`)
- `fromName: null`, `replyTo: null` (framework sent values — dropped because they're top-level, not nested)
- `content: { templatePath: "@hubspot/email/dnd/plain_text.html", widgets: {} }` — framework's `content.html` silently discarded

TSC's working email (`375454060744`) — the comparison target — is:
- `type: "AUTOMATED_EMAIL"`, `subcategory: "automated"`, `state: "AUTOMATED"`, `emailTemplateMode: "DRAG_AND_DROP"`
- `from: { fromName: "The Single Circle", replyTo: "tsc@digitalmercenaries.ai" }` — **nested `from` object**, not top-level
- `language: "en-gb"`
- `subscriptionDetails: { subscriptionId: "2269639338", officeLocationId: "375327044798", subscriptionName: "Marketing Information" }` — portal-specific IDs
- `content: { templatePath: "@hubspot/email/dnd/Plain_email.html", widgets: { primary_rich_text_module, preview_text, footer_module }, flexAreas: { main }, styleSettings: {...} }`
- `isPublished: true`, `isTransactional: false`

### Subscription ID discovery

To automate subscription lookup, framework would need the `communication_preferences.read` scope. We currently require 7 scopes; adding this one would change the scope list (v1.5+ territory, not this patch).

For v1.4.0, treat `subscription_id` and `office_location_id` as user-supplied inputs. User looks them up once in HubSpot UI (Settings → Marketing → Email → Subscription Types / Office Locations) and sets them in the account profile.

### New account profile variables

Add to `~/.config/hs-lander/<account>/config.sh` schema (framework scripts read the profile; module receives them as `TF_VAR_*`):

- `HUBSPOT_SUBSCRIPTION_ID` — required, e.g. `"2269639338"` (from HubSpot Subscription Types UI)
- `HUBSPOT_OFFICE_LOCATION_ID` — required, e.g. `"375327044798"` (from HubSpot Office Locations UI)
- `HUBSPOT_SUBSCRIPTION_NAME` — optional display name for reference, e.g. `"Marketing Information"` (informational, not sent to API)

The `tf.sh` script exports these as `TF_VAR_hubspot_subscription_id` and `TF_VAR_hubspot_office_location_id`.

### New module variables

`terraform/modules/landing-page/variables.tf`:

```hcl
variable "hubspot_subscription_id" {
  description = "HubSpot subscription ID for the welcome email. Look up in HubSpot UI: Settings → Marketing → Email → Subscription Types. Per-portal value."
  type        = string
}

variable "hubspot_office_location_id" {
  description = "HubSpot office location ID for the welcome email. Look up in HubSpot UI: Settings → Marketing → Email → Office Locations. Per-portal value."
  type        = string
}

variable "email_language" {
  description = "Email language code (e.g. en-gb, en-us, fr, de)."
  type        = string
  default     = "en-gb"
}

variable "email_body_html" {
  description = "Raw HTML to embed inside the welcome email's primary rich-text widget. Rendered within HubSpot's DnD plain email template."
  type        = string
  # Sourced from file(var.email_body_path) at module call in scaffold/terraform/main.tf
}
```

### New payload

```hcl
resource "restapi_object" "welcome_email" {
  path          = "/marketing/v3/emails"
  id_attribute  = "id"
  update_method = "PATCH"

  data = jsonencode({
    name              = var.email_name
    subject           = var.email_subject
    type              = "AUTOMATED_EMAIL"
    subcategory       = "automated"
    state             = "AUTOMATED"
    emailTemplateMode = "DRAG_AND_DROP"
    language          = var.email_language
    isPublished       = true
    isTransactional   = false

    from = {
      fromName = var.email_from_name
      replyTo  = var.email_reply_to
    }

    subscriptionDetails = {
      subscriptionId     = var.hubspot_subscription_id
      officeLocationId   = var.hubspot_office_location_id
    }

    content = {
      templatePath = "@hubspot/email/dnd/Plain_email.html"
      widgets = {
        primary_rich_text_module = {
          body = {
            rich_text = var.email_body_html
          }
        }
      }
    }

    to = {
      limitSendFrequency  = false
      suppressGraymail    = false
    }
  })
}
```

### Skill guidance

When creating a new account profile, the skill asks the user to look up Subscription ID and Office Location ID in HubSpot UI. The skill does *not* attempt to probe them — the scope isn't available by default. Document this in `references/prerequisites.md`.

### Sender verification note

Even with the right payload, HubSpot may reject sends if the `from.replyTo` address isn't a verified sender on the portal. That's a portal-level configuration (Marketing → Email → Settings → From Address) — outside the framework's control. Document in `docs/framework.md` as a prerequisite: "The reply-to address must be a verified sender on the HubSpot portal. If unverified, the email can still be created but won't send when the workflow triggers."

- `type = "AUTOMATED_EMAIL"`, `subcategory = "automated"`, `state = "AUTOMATED"`
- `subscriptionDetails.subscriptionId` matches the account profile value

## Fix 4 — Landing page routing: use domain-level primary-page association

**File:** `terraform/modules/landing-page/pages.tf`

### Evidence from live probes

Heard's landing page (`395236739258`) created as:
- `domain: "heard.digitalmercenaries.ai"` (what framework sent)
- `slug: "-temporary-slug-8895232b-..."` (UUID-mangled from the empty string we sent)
- `url: "http://heard.digitalmercenaries.ai/-temporary-slug-..."` (broken — at UUID URL, not https, domain not connected)

TSC's working landing page (`375360721122`) stores:
- `domain: ""` (**empty!**)
- `slug: ""` (empty)
- `url: "https://tsc-landing.digitalmercenaries.ai"` — correct root URL

Cross-referenced against `/cms/v3/domains`:

- `tsc-landing.digitalmercenaries.ai` is in the domains list with `isPrimaryLandingPage: true`, `isPrimarySitePage: true`, `isPrimaryEmail: true`
- `heard.digitalmercenaries.ai` is **not in the domains list at all** — the domain hasn't been connected in HubSpot yet

### Diagnosis

The framework's approach is wrong in two ways:

1. **Page should have empty `domain`, not a specific domain.** HubSpot routes pages via per-domain configuration (the `isPrimaryLandingPage` flag on the domain record), not via the page record's own `domain` field. Sending a specific domain with empty slug causes HubSpot to mangle the slug into a UUID placeholder.

2. **Domain connection is a prerequisite.** Without `heard.digitalmercenaries.ai` being connected in HubSpot's Domains settings (and marked `isPrimaryLandingPage: true`), there's no routing target. HubSpot stuffs pages at UUID slugs because it has nowhere to serve them at domain-root.

### Fix

`pages.tf` — `landing_page` resource:

```hcl
data = jsonencode({
  name         = var.page_landing_name
  slug         = ""
  domain       = ""       # <-- was var.domain; now empty to enable domain-level routing
  htmlTitle    = var.page_landing_title
  templatePath = var.template_path_landing
  state        = "PUBLISHED"
  language     = "en-gb"
})
```

Same change applies to `thankyou_page` resource for consistency (though thank-you at `/thank-you` worked with the non-empty domain; changing for symmetry and because thank-you routing also resolves via the domain-primary flag).

### Manual prerequisite — document clearly

After `terraform apply` creates the landing page (with empty domain), the user must:

1. **Connect the domain in HubSpot UI** (Settings → Website → Domains & URLs → Connect a domain) — this triggers SSL provisioning and adds the domain to the Domains API list
2. **Designate the landing page as Primary Landing Page for that domain** — per domain in HubSpot UI: `isPrimaryLandingPage: true` against the newly-created page record
3. **Designate the thank-you page as Primary Site Page** — `isPrimarySitePage: true`

All three steps are in the HubSpot UI. The SKILL.md Step 9 DEPLOY section should be updated to surface these as required post-deploy manual steps.

### Future work (out of scope for v1.4.0)

Automating the domain-primary-page association via the Domains API is feasible but requires:
- A separate Terraform resource that PATCHes `/cms/v3/domains/<domain-id>` with `primaryLandingPageId: <landing-page-id>`
- Looking up the domain-id from the domain name (extra GET before PATCH)
- Handling the case where the domain hasn't been connected yet (creates an error that must be retried after manual connection)

Track as a roadmap item in the HubSpot domain automation bucket (v1.3 / v1.7 territory).

### Acceptance criteria

After the fix and the manual HubSpot UI steps:

- Landing page record has `domain: ""` and `slug: ""`
- The domain record (`/cms/v3/domains`) for the project's domain has `isPrimaryLandingPage: true` with `primaryLandingPageId` pointing at the new page
- `https://<domain>/` (root) returns HTTP 200 serving the landing page HTML
- `https://<domain>/thank-you` returns HTTP 200 serving the thank-you page

### Test addition

`tests/test-deployment.sh` — assert HTTP 200 on `https://<domain>/` (root), not just on `/thank-you`. Add a pre-check that verifies `GET /cms/v3/domains/<domain-id>` shows `isPrimaryLandingPage: true` — if not, surface a clear error directing the user to the manual UI step.
