## Problem

A v1.7.0 → v1.7.0 migration on a downstream Starter project surfaced four small problems that don't justify a minor-version bump but do warrant a patch.

### Issue 1 — `tests/test-deployment.sh` EXIT trap aborts before v1.7.0's served-asset checks run

`tests/test-deployment.sh:49` (or thereabouts — exact line per current source) installs an EXIT trap that uses `rm -f` for paths created with `mktemp -d`:

```bash
trap 'rm -f "$LANDING_HTML_FILE" "$THANKYOU_HTML_FILE" "$LANDING_TOKENS_DIR" "$THANKYOU_TOKENS_DIR"' EXIT
```

`rm -f` errors on directory paths under `set -e`, and the trap fires partway through the script's execution path. The result: every test run aborts before reaching the v1.7.0-introduced served-asset section (CSS 200, `hub_generated/template_assets`, `/hs/scriptloader/<portal>.js`, `prefers-color-scheme` in served CSS, `templateType: page` annotation, `preview_text` widget). Consumers see "8/8 passed" — the *original* base tests — and miss the new ones entirely.

### Issue 2 — `include_bottom_cta` module variable is wired to nothing

v1.7.0 added `include_bottom_cta` to `terraform/modules/landing-page/variables.tf` with `default = true`. The plan explicitly documented (in v1.7.0 §Fix 8) that the variable is "advisory metadata" until a real templating layer ships, but the variable itself is still present in `variables.tf` and exported via `tf.sh`. Consumers reading `variables.tf` reasonably assume setting `include_bottom_cta = false` will hide the second `hbspt.forms.create` — but the scaffold templates hardcode it. So `false` is silently ignored.

The misleading variable should be removed for v1.7.1; if a future release adds a real templating layer, the variable comes back at that point with actual wiring.

### Issue 3 — v1.6.7 migration note overstates PATCH-strip behaviour

v1.6.7's CHANGELOG migration block says: *"PATCH cannot fix existing welcome_email — HubSpot strips flexAreas on PATCH."* Empirical evidence from the v1.7.0 migration round is more nuanced: PATCH **preserves** `flexAreas` when the request body contains the **complete** section/column envelope (including `path: null`, `breakpointStyles`, etc.). The first attempt that "stripped to `{main: {}}`" was sending an incomplete structure; subsequent complete-structure PATCHes survived intact.

This matters because the migration cost from v1.6.x → v1.6.7/v1.7.0 currently mandates `terraform taint` + recreate, which:
- Resets state from AUTOMATED → AUTOMATED_DRAFT (consumer must re-publish in HubSpot UI)
- Changes the email ID, breaking any form-follow-up wiring (consumer must re-wire)

If complete-structure PATCH actually works (the framework's `update_data` block now contains the complete structure post-v1.6.7), the migration could be a noisy-but-clean in-place update with no UI re-work. The CHANGELOG note should be corrected.

### Issue 4 — `scripts/upgrade-project-scripts.sh` (R9) still missing

Three releases now have required consumers to manually refresh their project-local `scripts/` directory:
- v1.5.0 added `scripts/preflight.sh`
- v1.6.6 changed `scripts/upload.sh` (endpoint path)
- v1.6.6 changed `scripts/upload.sh` again (multipart)

Each upgrade triggered the same dance:

```bash
rm -rf scripts && cp -r $FRAMEWORK_HOME/scripts scripts && chmod +x scripts/*.sh
```

R9 in `docs/roadmap.md` has tracked this since v1.5.0 ("operator ergonomics"). Time to ship a sanctioned helper.

## Design

### Fix 1 — `tests/test-deployment.sh` EXIT trap

Split the trap into separate `rm -f` (files) and `rm -rf` (directories), each on its own line to avoid compound-operator approval prompts in environments with strict shell rules:

```bash
cleanup() {
  rm -f "$LANDING_HTML_FILE"
  rm -f "$THANKYOU_HTML_FILE"
  rm -rf "$LANDING_TOKENS_DIR"
  rm -rf "$THANKYOU_TOKENS_DIR"
}
trap cleanup EXIT
```

Exact variable names per current source — if the trap captures different paths than the four above, adjust accordingly. The functional change is `rm -f` → `rm -rf` for any path created via `mktemp -d`.

After the fix, run a clean test pass on a known-good v1.7.0 deploy to confirm all served-asset checks now execute and report. The implementer should expect ~17 checks total per the v1.7.0 plan (7 base + 4 surgical-fix API + 6 served-page/scaffold).

### Fix 2 — Remove `include_bottom_cta`

Three edits:

- `terraform/modules/landing-page/variables.tf`: remove the `variable "include_bottom_cta" {...}` block.
- `scripts/tf.sh`: remove the `TF_VAR_include_bottom_cta` export line (if present).
- `scripts/set-project-field.sh`: remove `INCLUDE_BOTTOM_CTA` from the allow-list.
- `scaffold-project.sh`: remove the `# INCLUDE_BOTTOM_CTA=true` line from the project-profile stub.

The scaffold landing template continues to ship two `hbspt.forms.create` instances by default; consumers who want a single hero CTA delete the second instance from their project's `src/templates/landing-page.html`. This preserves the v1.7.0 behaviour without the misleading variable. A future v1.8.x or v1.9.0 may reintroduce the variable when scaffold conditionals are technically supported (separate plan).

### Fix 3 — v1.6.7 migration-note correction (docs-only)

Update `CHANGELOG.md` v1.6.7 section's migration block. Replace the absolute "PATCH cannot fix existing welcome_email" claim with the conditional truth:

```markdown
## Migration from v1.6.6 (corrected)

The welcome-email widget-shape fix takes effect on v1.6.7+ via either
PATCH or CREATE — but **PATCH only succeeds when the request body
contains the complete flexAreas envelope** (sections, columns,
breakpointStyles, path). Prior to v1.6.7, framework `update_data`
shipped an incomplete structure and PATCH silently stripped flexAreas
to {main: {}}. From v1.6.7 onward, `update_data` contains the
complete structure — so a plain `terraform apply` updates existing
welcome_email resources in place without taint or recreate.

If your existing welcome_email is stuck (body.html populated but
flexAreas empty) from a pre-v1.6.7 PATCH attempt, run:

    bash scripts/tf.sh apply

If apply alone doesn't repair the layout (HubSpot's PATCH may have
locked the resource into a state where re-applying the same payload
is a no-op), fall back to taint + recreate:

    bash scripts/tf.sh taint module.landing_page.restapi_object.welcome_email
    bash scripts/tf.sh apply

Note that recreate changes the welcome_email ID, requiring you to:
- Re-publish in HubSpot UI (recreated email goes to AUTOMATED_DRAFT)
- Re-wire any form follow-up email setting (Marketing → Forms →
  Follow-up tab) since it references the old ID

Plain in-place PATCH avoids both. Try it first.
```

This is purely a CHANGELOG.md edit — no code changes. The plan flags it for clarity because consumers reading older CHANGELOG entries during a future upgrade need accurate information.

### Fix 4 — `scripts/upgrade-project-scripts.sh`

New script in framework `scripts/`. Refreshes a project's local `scripts/` directory from `$FRAMEWORK_HOME/scripts/` while preserving project-specific overrides (none today, but future-proofing).

```bash
#!/usr/bin/env bash
# upgrade-project-scripts.sh — Refresh project-local scripts/ from the framework's
# scripts/ directory. Run after bumping ?ref=vX.Y.Z and before npm run setup.
#
# Usage:
#   bash $FRAMEWORK_HOME/scripts/upgrade-project-scripts.sh [/path/to/project]
#
# If the project path is omitted, uses $PWD.

set -euo pipefail

PROJECT_DIR="${1:-$PWD}"
FRAMEWORK_SCRIPTS_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

if [[ ! -d "$PROJECT_DIR/scripts" ]]; then
  echo "ERROR: $PROJECT_DIR/scripts does not exist. Is this a hs-lander project?" >&2
  exit 1
fi

echo "Refreshing scripts in $PROJECT_DIR/scripts from $FRAMEWORK_SCRIPTS_DIR"

# Backup existing scripts to a timestamped directory.
BACKUP_DIR="$PROJECT_DIR/scripts.bak.$(date +%Y%m%d-%H%M%S)"
cp -r "$PROJECT_DIR/scripts" "$BACKUP_DIR"
echo "  Backup: $BACKUP_DIR"

# Replace scripts/ contents (don't delete scripts/ itself in case the project
# has a non-script file there, e.g. README).
rm -f "$PROJECT_DIR"/scripts/*.sh
cp "$FRAMEWORK_SCRIPTS_DIR"/*.sh "$PROJECT_DIR/scripts/"
chmod +x "$PROJECT_DIR"/scripts/*.sh

# Copy lib/ subdirectory if present in framework (added in v1.7.0).
if [[ -d "$FRAMEWORK_SCRIPTS_DIR/lib" ]]; then
  rm -rf "$PROJECT_DIR/scripts/lib"
  cp -r "$FRAMEWORK_SCRIPTS_DIR/lib" "$PROJECT_DIR/scripts/lib"
fi

echo "Done. Run 'npm run tf:init -- -upgrade' next, then npm run setup."
echo "If anything breaks, restore from $BACKUP_DIR."
```

Header comment establishes:
- Backup is created before any change (timestamp-stamped, easy to restore from)
- Idempotent — re-running just refreshes again from the framework
- Doesn't touch `scripts/` non-script files (rare today, but possible future state)
- Copies the `lib/` subdirectory (added in v1.7.0 for `tier-classify.sh`)

CHANGELOG entry adds:

```markdown
### Added

- `scripts/upgrade-project-scripts.sh` — sanctioned helper for
  refreshing project-local scripts on framework version bumps.
  Replaces the manual `rm -rf scripts && cp -r $FRAMEWORK_HOME/scripts ...`
  dance that's been required since v1.5.0. Backs up existing scripts/
  to a timestamped scripts.bak.<ts>/ directory before replacing.
  Idempotent.
```
