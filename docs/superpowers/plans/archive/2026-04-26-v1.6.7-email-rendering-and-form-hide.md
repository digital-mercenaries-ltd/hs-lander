# Plan: v1.6.7 — Email rendering and form-field hide

**Date:** 2026-04-26
**Status:** Complete (shipped as v1.6.7 on 2026-04-26)
**Scope:** Framework. Surgical bug fixes only — no scaffold or preflight changes (those live in v1.7.0). Two source fixes plus the four narrow API-level tests that verify them.
**Target release:** v1.6.7 (patch)
**Dependency:** v1.6.6 shipped (which carried the `scripts/upload.sh` multipart fix per `2026-04-26-v1.6.6-upload-multipart.md`).

## Problem

A downstream deploy round surfaced two latent framework bugs that compound to "the welcome email never renders for any project that ever used the framework" and "the segmentation field is visible on the rendered form". A third bug from the same round (`scripts/upload.sh` Content-Type) shipped as v1.6.6 and is not in scope here.

### Issue 1 — Welcome email widget shape is wrong (compound: three bugs)

`terraform/modules/landing-page/emails.tf` has been writing welcome-email body content into a widget whose shape is silently rejected by HubSpot's render engine since v1.5.0. Three compounding problems:

1. **Wrong field key.** Body content is written to `widgets.primary_rich_text_module.body.rich_text`. HubSpot's render engine reads from `widgets.primary_rich_text_module.body.html`. The API accepts both writes silently — only renders the latter.
2. **Missing widget metadata.** The widget object is shipped as `{body: {...}}` only — no `id`, `name`, `module_id`, `type`, `order`. HubSpot can't classify a widget without metadata, so it's never rendered. A working email (compared empirically against an older project's working email) carries `module_id: 1155639` and `type: "module"` plus full top-level metadata.
3. **Missing layout placement.** The create payload has no `content.flexAreas`. HubSpot generates a default layout containing only `footer_module`. Even if the widget shape were correct, it has nowhere to render.

The `update_data` (PATCH) currently keeps reasserting `body.rich_text` on every apply. Since `flexAreas` is also stripped on PATCH (see `references/hubspot-api-quirks.md` v1.7.0), no amount of body edits will surface in the email until the layout placement is added on the *create* path.

### Issue 2 — `project_source` segmentation field is visible

The `project_source` capture form field exists for cross-project contact segmentation. Since v1.5.0+ it has been hidden via CSS only. HubSpot Forms v3 markup differs from v2, so `.hs_project_source` selectors miss in many browsers. The canonical field-hide mechanism — HubSpot's own `hidden: true` form-field flag — has not been set in `forms.tf`.

`/marketing/v3/forms/{id}` confirms the field currently ships with `hidden: false`. Setting the flag at field-definition time is HubSpot's documented way to hide a field from the rendered form regardless of class-name churn.

## Design

### Fix 1 — `emails.tf` widget shape

Apply to **both** `data` (POST) and `update_data` (PATCH) blocks of `restapi_object.welcome_email`.

```hcl
content = {
  widgets = {
    primary_rich_text_module = {
      body = {
        html                     = var.email_body_html
        hs_enable_module_padding = false
        module_id                = 1155639     # confirmed via Prerequisite A
      }
      id         = "primary_rich_text_module"
      name       = "primary_rich_text_module"
      module_id  = 1155639
      order      = 0
      type       = "module"
      label      = null
      smart_type = null
      child_css  = {}
      css        = {}
      styles     = {}
    }
  }
  flexAreas = {
    main = {
      # <<FLEXAREAS_FROM_PROBE>> — paste verbatim from Prerequisite B output.
      # Must include section + column envelope (style, breakpointStyles, path) and
      # column.widgets = ["primary_rich_text_module", "footer_module"]
    }
  }
}
```

`update_data` carries the same shape — PATCH preserves the `body.html` write but strips layout, hence the migration step (see Migration). v1.6.7 deliberately ships **without** the `preview_text` widget; that requires a new module variable and is v1.7.0 scope.

### Fix 2 — `forms.tf` hidden flag

In `terraform/modules/landing-page/forms.tf`, locate the `project_source` field definition in **both** `capture_form` and `survey_form` (the comment block was normalised in v1.6.2 so a single `replace_all` should hit both):

```hcl
{
  name         = "project_source"
  label        = "Project Source"
  fieldType    = "single_line_text"
  objectTypeId = "0-1"
  hidden       = true                       # <-- add this line
  defaultValue = var.project_slug
}
```

`hidden = true` is HubSpot's canonical field-hide flag, distinct from the deprecated `fieldType = "hidden"`. CSS-based hiding remains as defence-in-depth (lives in scaffold CSS — handled in v1.7.0), but the flag is the primary mechanism.

Header comment update: "`project_source` is hidden via the form-level `hidden: true` flag (HubSpot's canonical mechanism). Scaffold CSS adds belt-and-braces selectors for older portals; `fieldType = \"hidden\"` is deprecated and rejected on Forms v3 — do not revive it."

## Migration

This is the critical migration call: **PATCH cannot fix existing welcome_email resources.** The `/marketing/v3/emails` PATCH surface silently strips `flexAreas` to `{}` (confirmed empirically). So consumers upgrading from v1.6.6 to v1.6.7 will get `body.html` written via PATCH but **no layout placement**, leaving the email still empty.

Required upgrade sequence per consumer:

```bash
# In project directory:
bash scripts/tf.sh taint module.landing_page.restapi_object.welcome_email
bash scripts/tf.sh apply
```

The `taint` triggers a destroy + recreate; the `data` (POST) payload includes the full `flexAreas` shape, so the recreated email lands correct. `terraform_data.publish_welcome_email` re-fires automatically because its `triggers_replace` is keyed on the email ID.

**On Starter portals**, that re-fire will surface a `MISSING_SCOPES` error from HubSpot:

```
This app hasn't been granted all required scopes.
requiredGranularScopes: ["transactional-email", "marketing-email"]
```

This is **expected and harmless** — those scopes are tier-gated and unavailable on Starter regardless of Service Key configuration. The email itself is recreated correctly; only the auto-publish step fails. Manual UI publish (Marketing → Email → '<project> — Welcome' → Review and publish) completes the flow. v1.7.0 makes the publish step opt-out via the `auto_publish_welcome_email` variable so this error stops appearing on Starter.

CHANGELOG migration block must spell out both the taint and the Starter scope expectation:

