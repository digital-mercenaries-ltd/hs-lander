# Plan: v1.9.2 — Portal-shared resource protection (`project_source`)

**Date:** 2026-04-28
**Status:** Complete (shipped in v1.9.2, 2026-04-28). All three mitigations landed in the framework PR; skill follow-up tracked separately.
**Scope:** Framework + skill. Three additive mitigations that make accidental destruction of the portal-shared `project_source` CRM property structurally impossible. No module input contract changes; no breaking changes.
**Target release:** v1.9.2 (framework, patch). Skill bump tracked separately under the existing v1.9.x adoption work — this plan only enumerates the skill-side change; the skill PR is a follow-up.
**Dependency:** v1.9.1 shipped. The plan-review gate (v1.9.0) is the integration point for component 2.

## Problem

The framework creates a single shared CRM contact property — `project_source` — in `terraform/modules/account-setup/main.tf:17`. Every per-project landing-page deployment on the same HubSpot portal references it by name (`terraform/modules/landing-page/lists.tf:51` and `forms.tf:156,274`) for contact-list segmentation. The property itself lives only in the **account-setup** Terraform state.

Today there is no structural guard against destroying it:

- An operator who runs `terraform destroy` from the account-setup directory deletes the property. Every other project on the portal that filters its contact list on `project_source` is now orphaned — the list filter references a name that no longer exists, and re-applying any project recreates the property fresh, losing all historical segmentation tags written to existing contacts.
- The v1.9.0 plan-review gate already raises `PLAN_REVIEW=confirm severity=destructive` for any destroy, but a confirmable plan is still confirmable. The gate doesn't know that this particular delete has portal-wide blast radius.
- The skill's Step 9 deploy report shows `PLAN_RESOURCES.delete` to the operator but doesn't single out portal-shared resources for special handling.

The single live failure mode flagged in the turn-2389 risk audit (chat session `s3-hs-lander-skill-build`) is exactly this. Three layered guards close it.

## Mitigations

### 1 — Terraform `prevent_destroy` on `project_source_property`

Add a `lifecycle` block to `restapi_object.project_source_property` in `terraform/modules/account-setup/main.tf`:

```hcl
lifecycle {
  prevent_destroy = true
}
```

Effect: any plan that would destroy or replace this resource fails at plan time with Terraform's built-in `Resource cannot be destroyed` error. No `--force` flag exists; the operator must explicitly remove the lifecycle block to proceed. That enforces a deliberate, auditable two-step.

Properties:

- Pure metadata; no HubSpot API call on upgrade. Existing account-setup states absorb it on the next refresh with zero plan diff.
- Doesn't affect create/update — only destroy and replace.
- Doesn't restrict legitimate retirement: an operator decommissioning a portal removes the block first, which is the right ergonomics.
- Scope is correctly the account-setup module only. Per-project resources (capture form, contact list, welcome email, pages) remain freely destroyable, because retiring a single project should still destroy them.

### 2 — Plan-review `portal-shared` severity tier

Extend `scripts/plan-review.sh` so that when `delete_addrs` or `replace_addrs` includes any address matching a portal-shared allowlist, the emitted severity escalates above `destructive`.

Contract change (additive, backwards-compatible):

- `PLAN_REVIEW_SEVERITY` gains a new value `portal-shared`. Severity rank becomes `portal-shared > destructive > caution > info` (line 148 comment updates).
- Allowlist (initial): the substring `project_source_property` matched against each resource address. Documented in a constant near the existing `max_create` / `max_update` thresholds so the next portal-shared resource (if any) extends the list in one place.
- A new contract line `PLAN_REVIEW_PORTAL_SHARED=<csv>` lists the matching resource addresses, but only when the severity is `portal-shared`. Empty otherwise.

Even with mitigation 1 in place, this gate fires before Terraform errors out — it gives the skill a clean, machine-readable signal to abort with a tailored message rather than surfacing the raw Terraform error. Belt-and-braces.

Test additions:

- `tests/test-plan-review.sh` (new, if it doesn't exist; otherwise extend) — feed a synthetic plan JSON with `project_source_property` in `delete_addrs` and assert the new severity + contract line. Same fixture pattern as `tests/test-terraform-plan.sh`.

### 3 — Skill Step 9 abort + coaching block

Skill side (`~/DocsLocal/skills/hs-lander/SKILL.md`):

- Step 9 deploy flow: when `PLAN_REVIEW_SEVERITY=portal-shared`, abort the deploy. Surface the matching addresses from `PLAN_REVIEW_PORTAL_SHARED` and a fixed coaching block explaining the blast radius and the deliberate two-step the operator must take to proceed (remove `prevent_destroy`, re-plan, re-confirm).
- New coaching block in the framework-compatibility / safety section warning operators that `npm run destroy` against the **account-setup** directory orphans every project on the portal that filters on `project_source`. Cross-references the `prevent_destroy` guard so the operator knows what error they're hitting if they try.

This is the only behavioural change the skill needs. It doesn't change any existing flow — it adds a third terminal state to Step 9 alongside `ok` and `confirm`.

## Out of scope

- Adding `prevent_destroy` to per-project resources. They should remain destroyable; the project-retirement use case is real.
- A general-purpose "portal-shared resource registry" abstraction. Only one resource qualifies today; introduce the abstraction when the second one appears.
- Restoring a destroyed `project_source` from backup. The framework's `backup-file.sh` snapshots Terraform state but not HubSpot CRM properties; if the property is destroyed via UI or API outside Terraform, recovery is HubSpot's account-export path. Documented as a known gap in the coaching block, not solved here.
- Skill PR. Mitigation 3 is described here so the framework + skill stay co-designed, but the skill repo's own plan and PR are separate work — likely folded into the in-flight v1.9.x adoption session if it hasn't merged yet, otherwise a small follow-up.

## Risk + rollback

- **Existing deployed account-setup states** absorb the lifecycle change with zero plan diff. No HubSpot API calls. Rollback is removing the block — also a metadata-only change.
- **Plan-review consumers** that don't recognise the `portal-shared` severity will treat it as an unknown string. The skill currently switches on the value; it gains a new branch in mitigation 3. Any external consumer either gains the same branch or falls through to the existing `confirm` path — equivalent to today's behaviour. Backwards-compatible.
- **Operator who genuinely wants to retire a portal** removes `prevent_destroy` from the source, re-plans, and applies. The two-step is the intended ergonomics.

## Follow-ups

- If a second portal-shared resource is added (e.g. account-wide subscription type or sender domain owned by `account-setup`), extend the allowlist constant in `plan-review.sh` rather than re-architecting.
- The HubSpot account-export documentation gap — recovery if the property is destroyed via UI or API — belongs in a future operator-runbook plan, not this one.
