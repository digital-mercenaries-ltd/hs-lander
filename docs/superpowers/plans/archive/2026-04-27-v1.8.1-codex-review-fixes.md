# Plan: v1.8.1 — Review-and-deploy defects + documentation re-sync

**Date:** 2026-04-27
**Status:** Complete (shipped in v1.8.1, 2026-04-27)
**Scope:** Framework. Surgical bug fixes from three independent v1.8.0 reviews — a codex code review, an architectural review run in this session, and the Heard v1.8.0 deploy session — plus a documentation re-sync. No module input contract changes; no new module variables; no behaviour changes from the consumer's perspective beyond closing the dead-code paths and Starter-blocking defects the reviews surfaced.
**Target release:** v1.8.1 (patch — defects + docs).
**Dependency:** v1.8.0 shipped.

## Problem

Three independent reviews of v1.8.0 surfaced eleven defects and one documentation-drift issue:

- **Codex code review** (8 defects, P1 + P2).
- **Architectural review run in this session** (documentation drift compounding the contract issues).
- **Heard v1.8.0 deploy session** (3 defects exposed by real-world deploy on a Starter portal with `include_survey = true`; Heard worked around all three manually but they block any new Starter consumer).

None individually justify a minor bump; together they justify a patch. One of the defects (`EMAIL_REPLY_TO` unreachable) is a v1.8.0 feature that doesn't actually function — half of the email-DNS preflight is dead code in the realistic configuration path. Three more (B1 dropdown fieldType, B2 bool property options, B5 survey_form race) block first-apply for any new Starter project that uses `include_survey`.

The defects fall into four categories:

1. **Framework / scaffold contract drift** (P1) — what the framework says it does and what scaffolded projects actually receive don't match.
2. **Cross-script consistency gaps** (P2) — patterns that one script gets right and another gets wrong, with no shared library to hold the line.
3. **HubSpot API shape misses** (P1, from real-world deploy) — informed-guesses I shipped in v1.8.0 that real apply rejects.
4. **Lockfile / lint hygiene** (P2) — committed artefacts that have drifted past their declared constraints.

Plus the documentation drift, which compounds the contract issues for new contributors.

One further v1.8.0 defect (B3 — welcome_email PATCH fails once the email is in `AUTOMATED` published state) is **deferred** to its own plan because the fix is architecturally substantial (orchestrate unpublish → PATCH → republish via `terraform_data` local-exec). See `2026-04-27-welcome-email-published-state-handling.md` (stub) and Heard's interim workaround documented there.

### Issue 1 — Scaffold module pin stale at `?ref=v1.6.0` (P1)

`scaffold/terraform/main.tf` lines 60–64 pin both modules to `v1.6.0`:

```hcl
module "account_setup" {
  source = "git::https://github.com/digital-mercenaries-ltd/hs-lander//terraform/modules/account-setup?ref=v1.6.0"
}

module "landing_page" {
  source = "git::https://github.com/digital-mercenaries-ltd/hs-lander//terraform/modules/landing-page?ref=v1.6.0"
  ...
}
```

`VERSION` is `1.8.0`. Newly scaffolded projects copy v1.8.0 scripts and `VERSION` but their Terraform root applies v1.6.0 module code. Result: split-brain — the project's `tf.sh` exports `TF_VAR_*` for variables that v1.6.0 modules don't declare, and the v1.6.7 / v1.7.0 / v1.8.0 fixes (welcome email widget shape, `hidden=true` on project_source, preview_text widget, capture form redirect default, survey schema extensions) silently miss because the modules are old.

This has been broken since v1.6.0 — every release since has updated `VERSION` without updating the scaffold pin. The miss is invisible until a fresh scaffold is run.

### Issue 2 — `tf.sh` exports `TF_VAR_*` the scaffold root never declares or passes (P1)

`scripts/tf.sh:41-42` exports `TF_VAR_email_preview_text` and `TF_VAR_auto_publish_welcome_email` (added in v1.7.0 / v1.8.0). `set-project-field.sh`'s allow-list accepts `EMAIL_PREVIEW_TEXT` and `AUTO_PUBLISH_WELCOME_EMAIL` as project-profile keys. But `scaffold/terraform/main.tf` declares no matching root variables and passes nothing to `module.landing_page`. In a scaffolded project, setting these via `set-project-field.sh` writes the profile, `tf.sh` exports the value, Terraform sees an unused `TF_VAR_*` and silently ignores it. The skill's coaching says "to suppress publish on Starter, set `AUTO_PUBLISH_WELCOME_EMAIL=false`" — it has no effect.

This compounds Issue 1: even after the scaffold pin bumps to v1.8.1, the variables still aren't wired through unless the scaffold root declares them.

### Issue 3 — `EMAIL_REPLY_TO` is unreachable in `preflight.sh`'s `PREFLIGHT_EMAIL_DNS` block (P1)

v1.8.0 shipped `PREFLIGHT_EMAIL_DNS` with this preference order in `scripts/preflight.sh`:

```bash
if [[ -n "${EMAIL_REPLY_TO:-}" ]]; then
  _email_auth_domain="${EMAIL_REPLY_TO##*@}"
elif [[ -n "${DOMAIN:-}" ]]; then
  _email_auth_domain="$DOMAIN"
fi
```

`EMAIL_REPLY_TO` is never reachable in the realistic path:

- The `PROJECT_PROFILE` extractor at `preflight.sh:250` calls `_source_vars` for `PROJECT_SLUG`, `DOMAIN`, `DM_UPLOAD_PATH`, `GA4_MEASUREMENT_ID`, `CAPTURE_FORM_ID` only. It does not extract `EMAIL_REPLY_TO`.
- `scripts/set-project-field.sh`'s allow-list does not include `EMAIL_REPLY_TO`. A consumer cannot set it via the framework API.
- `scaffold-project.sh`'s project-profile stub does not seed it.

The `DOMAIN` fallback runs in every realistic case. Consumers whose email-sending domain differs from their landing-page domain (the common case once a project uses an email subdomain like `mail.example.com`) get the wrong DNS records validated. The check passes or fails against the wrong domain and the result is misleading either way.

This is a v1.8.0 bug that I shipped and missed in self-review. The fix is small but real.

### Issue 5 — Account/project argument validation is inconsistent (P2)

`accounts-init.sh:40-43` validates the account name with a strict regex:

```bash
if [[ ! "$account" =~ ^[a-z0-9][a-z0-9-]*$ ]]; then
  echo "ACCOUNTS_INIT=error invalid-account-name '$account' ..."
  exit 1
fi
```

This regex defeats path-traversal (`..`), uppercase, dots, slashes, and other characters that would corrupt the `~/.config/hs-lander/<account>/` filesystem layout.

Five other scripts that build paths from the same arguments do **not** validate:

- `scaffold-project.sh:25-27` (uses `$1`, `$2` directly to build paths under `accounts_dir`)
- `init-project-pointer.sh:20-21` (same)
- `set-project-field.sh:58-60` (same)
- `accounts-describe.sh:24-26` (uses `$1` directly)
- `projects-list.sh:20-22` (same)

A skill or operator passing `..` or a name with a slash to any of these constructs a path that escapes `~/.config/hs-lander/<account>/`. In practice the existing structure (Keychain-based credentials, atomic-mv writes, no-clobber checks) limits the blast radius — a `..` write would land outside the config dir but couldn't impersonate another account because Keychain entries are tied to service names, not paths. But the contract should be defended uniformly at the boundary.

### Issue 6 — `post-apply.sh` hardcodes `$HOME/.config` (P2)

`scripts/post-apply.sh:24` builds the target path as:

```bash
CONFIG_FILE="${HOME}/.config/hs-lander/${HS_LANDER_ACCOUNT}/${HS_LANDER_PROJECT}.sh"
```

Every other config-mutating script honours `HS_LANDER_CONFIG_DIR`:

```bash
accounts_dir="${HS_LANDER_CONFIG_DIR:-$HOME/.config/hs-lander}"
```

The inconsistency breaks the framework's test-sandboxing pattern. `tests/test-post-apply.sh` works around it by setting `HOME` to a temp directory, but no other test script needs that workaround. Future tests that exercise `post-apply.sh` against a sandboxed config dir would have to repeat the `HOME` override.

Additionally, `post-apply.sh` uses `_sed_inplace` to update existing assignment lines but has no fallback for missing lines. If a project profile is hand-edited and a line is removed (e.g. `SURVEY_FORM_ID` deleted because the project doesn't use a survey), `post-apply.sh` reports success but writes nothing for that field. Lower-impact than Issue 5; worth fixing in the same patch.

### Issue 7 — `build.sh` sed substitution doesn't escape replacement metacharacters (P2)

`scripts/build.sh` uses `|` as the sed delimiter (chosen to avoid escaping forward slashes in `DM_UPLOAD_PATH`):

```bash
_sed_inplace \
  -e "s|__DOMAIN__|${DOMAIN}|g" \
  -e "s|__DM_PATH__|${DM_UPLOAD_PATH}|g" \
  ...
```

If a config value contains `|`, `&`, or `\`, sed interprets them as metacharacters: `|` ends the substitution, `&` inserts the matched pattern, `\` starts an escape. `set-project-field.sh` rejects `\`, `"`, `$`, backtick, and control characters but explicitly accepts `|` and `&` (banned-char list at `set-project-field.sh:84`). A `DOMAIN=foo|bar` value passes set-project-field's validation, then corrupts the build output.

In practice, no realistic domain contains `|` or `&`. The miss is theoretical for the configured fields today. It becomes practical the moment any new substituted token can carry user-supplied free text (e.g. `__BRAND_NAME__` if it ever moves into `build.sh`).

### Issue 8 — CI shellcheck doesn't recurse into `scripts/lib/` (P2)

`.github/workflows/ci.yml:18` globs `scripts/*.sh tests/*.sh`. `scripts/lib/tier-classify.sh` (added in v1.7.0) and any future `scripts/lib/*.sh` files are unlinted. Today `scripts/lib/` has one file; v1.9.0's planned consolidation will land several more. Better to fix the glob now than discover the gap when six unlinted files have already shipped.

### Issue 10 — `dropdown` fieldType wrong in survey form rendering (B1, P1)

`scripts/preflight.sh`'s `_survey_field_rendered` block in `terraform/modules/landing-page/forms.tf` (added in v1.8.0) maps:

```hcl

  f.type == "single_line_text"     ? "single_line_text" :
  f.type == "dropdown"             ? "single_line_text" :    # ← informed-guess; wrong
  f.type == "multiple_checkboxes"  ? "multiple_checkboxes" :
  f.type == "radio"                ? "radio" :

```

I shipped `dropdown → "single_line_text"` in v1.8.0 with a TODO comment flagging Prerequisite-A verification. Heard's deploy hit it: HubSpot Forms v3 rejects the create with a fieldType error. The correct mapping is `dropdown → "dropdown"`.

Lower urgency than the others below because v1.8.0's design Path B (static thank-you form + Forms Submissions API) bypasses the embedded HubSpot survey form entirely — a project that uses `<select>` static markup for its survey doesn't trigger this bug. But any project that chooses to embed the HubSpot-defined survey form via `hbspt.forms.create` (Path A in the v1.8.0 plan's "phantom-form" decision) hits it.

### Issue 11 — `bool` properties created without canonical options (B2, P1)

`terraform/modules/landing-page/properties.tf` (v1.8.0) emits the property `data` payload with the `options` key only when `type == "enumeration"`:

```hcl
each.value.type == "enumeration" ? {
  options = [for i, opt in each.value.options : {label = opt, value = opt, displayOrder = i, hidden = false}]
} : {},
```

For `type == "bool"`, no options are emitted. HubSpot CRM API for boolean properties **requires** the canonical True/False options to be declared at create time:

```json
"options": [
  {"label": "True",  "value": "true",  "displayOrder": 0, "hidden": false},
  {"label": "False", "value": "false", "displayOrder": 1, "hidden": false}
]
```

Without them, the API returns 400 on create. The auto-added `<project_slug>_survey_completed` property (introduced in v1.8.0 when `include_survey = true`) hits this on every fresh Starter deploy. Heard worked around it manually (created the property via API + `terraform import`); future Starter consumers with `include_survey = true` will fail-on-first-apply until this fix lands.

The bug is pure misreading of the HubSpot CRM properties API contract: the original comment in v1.8.0 said *"HubSpot rejects {options: []} on non-enumeration types but accepts the key being entirely absent"* — true for `string` / `number` / un-typed properties, but `bool` is the exception that needs the options array populated with the True/False shape.

### Issue 12 — `survey_form` races `custom_property` creation (B5, P1)

`terraform/modules/landing-page/forms.tf` (v1.8.0) declares `restapi_object.survey_form` with no explicit dependency on the per-project `custom_property` resources. Terraform parallelises the apply, so when survey field names match custom property names (the normal case — survey fields write into matching CRM properties), HubSpot tries to create the form referencing properties that don't yet exist on the contact schema. The form creation 400s.

Heard worked around this by running `terraform apply` twice — first apply fails on form creation but succeeds on properties; second apply succeeds because properties now exist. Any consumer hitting it the first time without that knowledge sees the failure and assumes the framework is broken.

The fix is a one-line `depends_on` declaration on the survey form, pointing at the properties it semantically references.

### Fix 5 — Centralised account/project name validation

Introduce a single validator function in a shared location and consume it from every script that builds filesystem paths from account/project arguments. This is a small lib extraction; v1.9.0's broader `scripts/lib/` consolidation will pull more helpers in, but the validator is small enough to land in the patch without dragging the larger refactor along.

**New file `scripts/lib/validate-name.sh`:**

```bash
#!/usr/bin/env bash
# validate-name.sh — defensive validation for account / project names that
# build filesystem paths under ~/.config/hs-lander/<account>/<project>.sh.
#
# Sourced by accounts-init.sh, scaffold-project.sh, init-project-pointer.sh,
# set-project-field.sh, accounts-describe.sh, projects-list.sh.

# Validates that a string is a valid account or project name:
# lowercase letters, digits, hyphens; must start with letter or digit.
# Defeats: path traversal (..), uppercase, dots, slashes, spaces, control
# chars. The pattern matches accounts-init.sh's existing regex, which has
# been the de-facto standard since v1.3.0.
#
# Usage:
#   if ! is_valid_name "$account"; then
#     echo "<COMMAND>=error invalid-account-name '$account' (...)"
#     exit 1
#   fi
is_valid_name() {
  [[ "$1" =~ ^[a-z0-9][a-z0-9-]*$ ]]
}
```

**Update each consumer** to source the lib and call the validator early, before any path is constructed from the argument. Each script's existing error-output prefix (`SCAFFOLD=`, `INIT_POINTER=`, `SET_FIELD=`, `ACCOUNT_STATUS=`) is preserved — the validator returns boolean; the calling script formats the error.

Example for `scaffold-project.sh`:

```bash
# shellcheck source=lib/validate-name.sh
source "$(dirname "${BASH_SOURCE[0]}")/lib/validate-name.sh"

if ! is_valid_name "$account"; then
  echo "SCAFFOLD=error invalid-account-name '$account' (expected lowercase letters, digits, hyphens)"
  exit 1
fi

if ! is_valid_name "$project"; then
  echo "SCAFFOLD=error invalid-project-name '$project' (expected lowercase letters, digits, hyphens)"
  exit 1
fi
```

Same pattern for the other four scripts. `accounts-init.sh` keeps its existing inline regex initially — once the lib helper is in place and tested, it can switch to `is_valid_name` in a follow-up cleanup.

**Tests:** add a scenario per script asserting that invalid names (with `..`, `/`, uppercase, etc.) are rejected before any filesystem operation. For `accounts-describe.sh` and `projects-list.sh`, this is new test surface.

### Fix 7 — `build.sh` sed-escape replacement values

Add a small escape helper before the substitution call:

```bash
_sed_escape() {
  # Escape sed replacement metacharacters: |, &, and \.
  # Values cleared by set-project-field.sh's banned-char check still allow
  # | and &, so escape defensively.
  printf '%s' "$1" | sed -e 's/[\\|&]/\\&/g'
}

_HUBSPOT_PORTAL_ID=$(_sed_escape "$HUBSPOT_PORTAL_ID")
_HUBSPOT_REGION=$(_sed_escape "$HUBSPOT_REGION")
# ... and for every other substituted value
```

Then substitute the escaped values into the sed pattern. The token names (left side) don't need escaping because they're hardcoded. Only replacement values (right side) need it.

Alternative (rejected): switch sed delimiter to a less-likely-to-collide character (`\x01` etc.). Brittle and hostile to read; the escape function is explicit and inspectable.

### Fix 8 — CI shellcheck globs `scripts/lib/`

`.github/workflows/ci.yml:18`:

```yaml
- name: ShellCheck
  run: shellcheck scripts/*.sh scripts/lib/*.sh tests/*.sh
```

If `scripts/lib/` doesn't exist (it does, but defensively), shellcheck errors. Use a glob that handles the empty case:

```yaml
- name: ShellCheck
  run: |
    shellcheck scripts/*.sh tests/*.sh
    if compgen -G 'scripts/lib/*.sh' > /dev/null; then
      shellcheck scripts/lib/*.sh
    fi
```

Or simpler — `shellcheck scripts/**/*.sh tests/*.sh` with `shopt -s globstar` if bash 4+ is available on the runner (it is — GitHub Actions Ubuntu runners ship bash 5+).

Pick one; both are fine. The simpler form is `shellcheck scripts/*.sh scripts/lib/*.sh tests/*.sh` since `scripts/lib/` is now a permanent directory.

### Fix 11 — `bool` property options (B2)

`terraform/modules/landing-page/properties.tf`. Extend the conditional in the `data = jsonencode(...)` block to handle `type == "bool"`:

```hcl
data = jsonencode(merge(
  {
    name      = each.value.name
    label     = each.value.label
    type      = each.value.type
    fieldType = each.value.fieldType
    groupName = each.value.groupName
  },
  each.value.type == "enumeration" ? {
    options = [
      for i, opt in each.value.options : {
        label        = opt
        value        = opt
        displayOrder = i
        hidden       = false
      }
    ]
  } : each.value.type == "bool" ? {
    options = [
      { label = "True",  value = "true",  displayOrder = 0, hidden = false },
      { label = "False", value = "false", displayOrder = 1, hidden = false },
    ]
  } : {},
))
```

Update the comment above the conditional to record what's now known: HubSpot CRM rejects `{options: []}` on `string` / `number` types and accepts the key being absent; `enumeration` requires the populated array; `bool` requires the canonical True/False array. The original comment was right for two cases out of three; v1.8.1 makes it right for all three.

### Fix 12 — `survey_form` depends_on `custom_property` (B5)

`terraform/modules/landing-page/forms.tf`. Add a `depends_on` to `restapi_object.survey_form`:

```hcl
resource "restapi_object" "survey_form" {
  count         = var.include_survey ? 1 : 0
  path          = "/marketing/v3/forms"
  id_attribute  = "id"
  update_method = "PATCH"

  depends_on = [restapi_object.custom_property]

  data = jsonencode({
    # ... existing payload ...
  })
}
```

Terraform's `depends_on` accepts a resource address; for a `for_each` resource, `restapi_object.custom_property` resolves to all instances. The form will wait until every property has finished creating before its own create runs. Eliminates the two-pass apply Heard worked around.

`capture_form` doesn't need the same `depends_on` — its fields are the email gateway plus user-defined `capture_form_fields` (typically not custom CRM properties; usually email + name + similar built-ins). If a future consumer passes `capture_form_fields` referencing custom properties, the same depends_on should be added there. v1.8.1 doesn't pre-emptively add it on the assumption that current usage is bounded; a follow-up patch can extend if needed.

## Migration

All v1.8.1 changes are non-breaking:

- Scaffold pin bump (Fix 1) and root-variable additions (Fix 2) only affect newly-scaffolded projects. Existing projects with `?ref=v1.6.0` or any other older pin keep working unchanged.
- `EMAIL_REPLY_TO` reachability (Fix 3) is additive — projects that don't set it continue to use the `DOMAIN` fallback.
- Lockfile regen (Fix 4) only matters for `terraform init` against the account-setup module directory.
- Validator (Fix 5) tightens an under-defended boundary. Names that were valid before remain valid (the regex matches `accounts-init.sh`'s existing rule). A name that *was* getting through the other scripts unchecked but *would* have been rejected by `accounts-init.sh` now gets rejected uniformly. No legitimate consumer should be affected.
- `post-apply.sh` and `build.sh` fixes (6, 7) are defects; before-fix behaviour was either broken or fragile.
- CI lint (8) is internal.
- Docs (9) update text only.

`terraform plan` against v1.8.1 should show zero changes for a v1.8.0 project. To upgrade:

```bash
bash $FRAMEWORK_HOME/scripts/upgrade-project-scripts.sh
# bump ?ref=v1.8.0 → ?ref=v1.8.1 in terraform/main.tf
bash scripts/tf.sh init -upgrade
bash scripts/tf.sh plan
```

For projects that want to use `EMAIL_REPLY_TO` for email-DNS preflight on a domain different from `DOMAIN`, run `bash $FRAMEWORK_HOME/scripts/set-project-field.sh <account> <project> EMAIL_REPLY_TO=hello@mail.example.com` once. Otherwise the existing `DOMAIN`-fallback behaviour is unchanged.

**Heard-specific note.** Heard's v1.8.0 deploy worked around B1, B2, and B5 manually:
- B1 is moot for Heard — `/thank-you` uses static `<select>` markup, never embeds the HubSpot survey form.
- B2 was worked around: `heard_survey_completed` was created via direct API call and `terraform import`-ed into state. Future plans on it will be in-place no-ops; the v1.8.1 fix means a *re-import after destroy* would create cleanly, but the current state isn't disturbed.
- B5 was worked around with two-pass apply. The v1.8.1 `depends_on` change adds a graph edge; existing state continues to work, and any future `terraform apply` from clean state runs in one pass.

So Heard sees zero state changes from B1/B2/B5 fixes. Net effect on Heard's project: cleaner `terraform plan` future-runs, no operational steps required.
