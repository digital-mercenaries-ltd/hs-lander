# Plan: v1.6.6 — upload.sh multipart fix

**Date:** 2026-04-26
**Status:** Complete (shipped as v1.6.6 on 2026-04-26)
**Scope:** Framework. One-file fix to `scripts/upload.sh` so `npm run deploy` actually transfers file bodies to HubSpot CMS Source Code v3.
**Target release:** v1.6.6 (patch — single script edit; no Terraform changes, no module-input changes)
**Dependency:** v1.6.5 shipped

## Problem

After v1.6.5 fixed the endpoint path (`/developer/content` → `/published/content`), Heard re-ran `npm run deploy` and got 8/8 uploads accepted with 2xx codes — but the templates didn't actually appear in Design Manager and the live pages continued serving HubSpot's "template not found" fallback. Heard's session diagnosis (paraphrased): the request reaches the right endpoint but the file body is not being read by HubSpot.

`scripts/upload.sh:47-51` currently sends:

```bash
curl -s -o /dev/null -w "%{http_code}" -X PUT \
  "${API_BASE}${dm_path}" \
  -H "Authorization: Bearer ${HUBSPOT_TOKEN}" \
  -H "Content-Type: application/octet-stream" \
  --data-binary "@${file}"
```

The CMS Source Code v3 API accepts `multipart/form-data` with a `file` part on `PUT /content/{path}`, not a raw octet-stream body. With the wrong content type, the endpoint may return 200/204 (the path is recognised) but treat the body as empty — hence the "ghost upload" symptom Heard observed: status codes look fine, files don't land. Heard locally patched their project's `scripts/upload.sh` to use multipart and confirmed 8/8 templates land in Design Manager.

## Design

Single-file edit to `scripts/upload.sh`. Replace the explicit `Content-Type` header + `--data-binary "@${file}"` with `-F "file=@${file}"`. curl autogenerates the `multipart/form-data; boundary=...` header from `-F`; an explicit `Content-Type` would clobber the boundary and reintroduce the bug.

Resulting block:

```bash
http_code=$(curl -s -o /dev/null -w "%{http_code}" -X PUT \
  "${API_BASE}${dm_path}" \
  -H "Authorization: Bearer ${HUBSPOT_TOKEN}" \
  -F "file=@${file}")
```

Header comment gains a one-line note:

```
# v3 expects multipart/form-data with a 'file' part. Do not add an explicit
# Content-Type header — curl's auto-generated boundary header would be lost.
```

**Alternative considered** — keep `--data-binary` and switch `Content-Type` to `multipart/form-data` manually. Rejected: requires hand-rolling the multipart boundary; curl's `-F` does it correctly and is the canonical pattern.

**Alternative considered** — switch to `application/json` with a base64-encoded body (HubSpot also documents a JSON shape for some endpoints). Rejected: the multipart shape is what Heard proved works against the live endpoint; introducing a JSON path would be speculative.
