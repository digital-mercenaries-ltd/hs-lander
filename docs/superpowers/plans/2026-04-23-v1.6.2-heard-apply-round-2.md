# Plan: v1.6.2 — Heard apply round 2 drift fixes

**Date:** 2026-04-23
**Status:** Pending
**Scope:** Framework. Three fixes surfaced when Heard applied against framework v1.6.1; plus a documented recovery path for pre-v1.6.0 emails stuck in the legacy payload shape.
**Target release:** v1.6.2 (patch — source fixes to existing modules, no contract change)
**Dependency:** v1.6.1 shipped

## Problem

Heard's 2026-04-23 re-deploy against v1.6.1 cleared the provider-version blocker and v1.6.0's list-resource work (`contact_list` created, listId 22). Three further errors remained:

1. **`survey_form` still uses `fieldType = "hidden"`.** v1.6.0 was supposed to convert both `capture_form` and `survey_form` from `"hidden"` to `"single_line_text"`. The edit only reached `capture_form` (line 62); `survey_form` at line 120 still carries the legacy value. Apply errors with the same `Could not resolve type id 'hidden'` message v1.6.0 fixed for `capture_form`. Root cause of the miss: the two field blocks have different leading comments, so a single `replace_all` edit match slipped by the survey variant.

2. **Forms v3 enum tightening on `richTextType`.** HubSpot v3 now requires `richTextType` to be one of `[image, text]`. Both `capture_form` and `survey_form` declare `richTextType = "NONE"`. Previously accepted; now rejected with `Invalid input JSON on line 1, column 480: Enum type must be one of: [image, text]`. Was masked by the earlier `fieldType = "hidden"` error in v1.6.0 probes; only surfaced now that the first error is out of the way.

3. **`landing_page` PATCH hits `PAGE_EXISTS` when slug="" is sent.** The landing-page resource sends `slug`, `domain`, `state` on every PATCH. Heard's landing page already exists on the portal with its correct domain; the PATCH to slug="" (root of domain) falls through HubSpot's primary-system-domain resolution, collides with TSC's long-lived root page (CONTENT_ID 375360721122 at `tsc-landing.digitalmercenaries.ai/`), and is rejected. Apply never gets to reconcile the real change (a name/title tweak), and the page stays on a HubSpot-generated temporary slug.

   The same update-immutability lesson applies: `slug`, `domain`, and `state` are effectively create-time fields. Mirroring the v1.6.0 welcome-email pattern, PATCH should omit them.

4. **Pre-v1.6.0 welcome emails are stuck in the old payload shape.** v1.6.0 moved the create payload to `AUTOMATED_EMAIL` / `state = AUTOMATED` / `isPublished = true` and deliberately omitted those fields from `update_data` so PATCH passes. Heard's email was created under v1.0.0 as `BATCH_EMAIL` / `DRAFT` / `isPublished = false`. Under v1.6.1 the editable fields updated cleanly (`from`, `language`, `subscriptionDetails`) but the three state-class fields cannot be promoted via PATCH — by v1.6.0 design, they're not in the PATCH payload. No framework bug here, but the framework has no documented recovery path, so consumers with existing emails are stuck.

## Design

### Fix 1 — Forms `survey_form` `fieldType`

One-line edit: `terraform/modules/landing-page/forms.tf:120` — `"hidden"` → `"single_line_text"`. Mirrors the capture_form edit shipped in v1.6.0. Also add the comment block above the survey_form block to match capture_form's (so a future `replace_all` doesn't miss it).

### Fix 2 — Forms `richTextType`

`terraform/modules/landing-page/forms.tf:29` and `:92` — `richTextType = "NONE"` → `richTextType = "text"`.

`"text"` is the safe default ("this field group uses rich-text metadata" is a no-op when no rich-text content is provided). The `"image"` option is for image-led field groups and doesn't apply here. Dropping the field entirely is not a safe option — HubSpot's v3 Forms API flags it required in the v3 schema even though the underlying data model does not use it for our field groups.

Header comment on the file gains a note explaining the enum constraint so future contributors don't revert to `"NONE"` from an older example.

### Fix 3 — Pages: split POST and PATCH payloads

Mirrors the v1.6.0 welcome-email fix. Both `landing_page` and `thankyou_page` resources in `terraform/modules/landing-page/pages.tf`:

- `data` (POST) keeps all fields: `name`, `slug`, `domain`, `htmlTitle`, `templatePath`, `state`.
- `update_data` (PATCH) sends only the editable fields: `name`, `htmlTitle`, `templatePath`.

Why not also omit `state` from POST: we want fresh pages to land published, same reasoning as the email. The `state` transition issue the v1.6.0 email hit doesn't apply to pages in the same way — HubSpot accepts `state = "PUBLISHED"` on page POST. But changing it on PATCH can collide with the URL-conflict check, so we drop it from `update_data`.

`slug` and `domain`: these are effectively identity for a page. Renaming a slug or moving a page between domains is supported in HubSpot but requires deliberate action (and often UI-level confirmation). Projects wanting to change these should do so via the HubSpot UI or via `terraform taint` + recreation, not via a silent in-place PATCH that may collide with other pages in the same domain-primary resolution.

Header comment on `pages.tf` gains a note explaining the split.

### Fix 4 — Pre-v1.6.0 email recovery (documentation only)

No code change. Add a section to `docs/framework.md` and to `CHANGELOG.md` v1.6.2 migration notes:

> **Stuck pre-v1.6.0 welcome emails.** Emails created by framework ≤ v1.5.0 have `type = "BATCH_EMAIL"` / `state = "DRAFT"` / `isPublished = false`. v1.6.0+ deliberately omits those fields from the PATCH payload (HubSpot's `/marketing/v3/emails/{id}` PATCH rejects transitions on them). Running `terraform plan` against such an email shows editable-field updates but no state transition. To promote the email to the current shape:
>
> ```bash
> # From the project root
> bash scripts/tf.sh taint module.landing_page.restapi_object.welcome_email
> npm run tf:plan    # confirms a CREATE action on the email resource
> npm run setup      # recreates the email with AUTOMATED_EMAIL / AUTOMATED / isPublished=true
> ```
>
> `taint` marks the resource for destroy+recreate on the next apply. The resulting email has the correct type/state/subcategory/isPublished and matches what a fresh v1.6.0+ apply would produce. A workflow that was attached to the old email ID in HubSpot must be re-attached to the new email ID (workflow binding is manual and not managed by Terraform).

Cleaner alternative considered — a `null_resource` that POSTs to `/marketing/v3/emails/{id}/publish` after create, with a `replace_triggered_by` guard keyed on a `data.external` check of the email's current type. Rejected for now: adds complexity for a one-time migration cost that only affects consumers on v1.5.0 or earlier with existing emails. Documentation + taint is adequate. If a future consumer hits this pattern repeatedly (e.g. a v2.0.0 migration), revisit.

## Scope limits

- **Workflow binding.** Still manual. Out of scope for v1.6.x; a roadmap item if it becomes a common enough pain point.
- **Page slug/domain migration.** The framework PATCH no longer sends these. Consumers wanting to change them use `terraform taint` + recreation, or the HubSpot UI. A first-class `migrate-page-slug` command could live on the skill side later.
- **Primary-landing-page-domain setting.** A separate R8 preflight can detect when the project's `DOMAIN` isn't connected or isn't set as primary. That work is already planned; unrelated to this patch.
