## Problem

Heard's re-deploy against v1.5.0 on portal 147959629 hit three distinct apply failures. The v1.5.0 apply had cleared earlier fixes (forms `formType`, property/list race, email payload restructure) but exposed three previously-hidden drift issues:

1. **Welcome email PATCH rejected.** On update, HubSpot returned `Cannot schedule or publish an email via the update API. Use the publish API instead.` The PATCH payload included `state = "AUTOMATED"` and `isPublished = true`, which HubSpot forbids toggling via the update API. The same fields were required on POST for the email to publish at creation. Single shared `data` block meant both paths used the same payload.
2. **Forms hidden field type removed.** On create, HubSpot returned `Could not resolve type id 'hidden' as a subtype of FieldBase; known type ids = [datepicker, dropdown, email, file, mobile_phone, multi_line_text, multiple_checkboxes, number, payment_link_radio, phone, radio, single_checkbox, single_line_text].` The `project_source` segmentation field used `fieldType = "hidden"` â€” valid in earlier Forms versions, removed from v3.
3. **Lists API wraps response.** On create, restapi provider raised `internal validation failed; object ID is not set` and left an orphan list (listId 21) on the portal. POST `/crm/v3/lists` returns `{"list": {"listId": ..., ...}}` with the actual payload nested under `list`; provider looked for `listId` at the response root and failed.

## Design

### Fix 1 â€” Email: split POST and PATCH payloads

Mastercard/restapi's `restapi_object` supports `update_data` as a distinct JSON body used for PATCH (falls back to `data` when unset). Source: `docs/resources/object.md` schema.

- `data` (POST) keeps the full payload including `state = "AUTOMATED"`, `isPublished = true`, `type = "AUTOMATED_EMAIL"`, `subcategory = "automated"`, `emailTemplateMode = "DRAG_AND_DROP"` â€” so creation publishes the email.
- `update_data` (PATCH) omits those five fields. Editable fields only: `name`, `subject`, `language`, nested `from`, `subscriptionDetails`, `content.widgets.primary_rich_text_module.body.rich_text`, `to` flags.

Alternative rejected: post-create publish via `null_resource` + `local-exec` hitting `POST /marketing/v3/emails/{id}/publish`. More moving parts; `update_data` is a cleaner single-file change.

### Fix 2 â€” Forms: single_line_text + scaffold CSS hide

Forms v3 FieldBase schema does not expose a `hidden: true` boolean or a "hidden" fieldType subtype. The `project_source` field is required for contact-list segmentation, so it must exist on submission but must not render to visitors.

- `fieldType` changed from `"hidden"` to `"single_line_text"` on both `capture_form` and `survey_form`. `defaultValue = var.project_slug` is retained so submissions carry the tag.
- Scaffolded `src/css/main.css` must hide the field. Requirement documented in `docs/framework.md`:
  ```css
  .hs-form input[name="project_source"],
  .hs-form .hs_project_source { display: none !important; }
  ```

Alternatives rejected:
- Drop the field entirely and inject via embed-time DOM manipulation or Forms Submit API. Requires skill-side changes and departs from the current "form definition is authoritative" pattern.
- Post-submit workflow tagging. Coupling segmentation to workflow execution is fragile.

### Fix 3 â€” List: nested id_attribute + suppress drift

The Mastercard/restapi provider's `id_attribute` supports slash-delimited paths via `internal/apiclient/utils.go: GetObjectAtKey`, which uses `strings.Split(path, "/")` and navigates nested maps. Confirmed by reading provider source.

- `id_attribute = "list/listId"` â€” extracts the ID from `response.list.listId` on POST and on subsequent GETs.
- `ignore_all_server_changes = true` â€” GET `/crm/v3/lists/{id}` returns the same wrapped shape, so field-by-field comparison of the flat `data` payload vs. the nested read response would falsely flag every field as drifted. Lists have minimal editable state (filter changes typically require replacement); suppressing drift is acceptable.

Alternatives rejected:
- `null_resource` + `local-exec` driving the list via hs-curl.sh, with listId captured to a file and read via `data.local_file`. Seven more moving parts than the two-line provider-config change. Appropriate fallback if the provider's nested-path support regresses.
- `read_search` with `search_patch` (RFC 6902). `read_search` is designed for list-search endpoints (using `results_key` to locate an array), not direct GET by id. Doesn't apply cleanly here.

## Scope limits (explicitly NOT in this plan)

- Publish-via-API for AUTOMATED emails. The email resource creates a send-ready email; attaching it to a form-submission workflow is still a HubSpot UI step. Deferred (roadmap candidate if it recurs).
- Filter-change handling on lists. Editing `filterBranch` after creation requires list replacement; `ignore_all_server_changes` means Terraform will not surface this. Acceptable trade-off; document if it causes confusion.
- Scaffold CSS generator. The skill generates `src/css/main.css` per project; updating the skill to include the `project_source` hide rule is a skill-side follow-up, not framework work.
