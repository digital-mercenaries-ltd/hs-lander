# Plan: First-Class Hosting Modes via `LANDING_SLUG`

**Date:** 2026-04-22
**Status:** Pending
**Scope:** Framework. Small additions to the landing-page module and scaffold template to support hosting modes beyond "connected custom domain". Landing-page module change only; no breaking change (new variable with a sensible default).
**Target release:** could land with v1.4.0 or as a follow-up v1.4.1. Independent of the v1.4.0 drift fixes but benefits from being in the same scaffold sweep.
**Context:** The Heard deploy (2026-04-22) exposed that the framework implicitly assumed one hosting mode — connected custom subdomain with pages at root via `isPrimaryLandingPage` flag. HubSpot Content Hub Starter limits accounts to one custom subdomain. The framework already hard-codes `slug = ""` in pages.tf (post-v1.4.0), which only works in the custom-domain mode. To support operators who need to use the HubSpot system domain or external redirect/iframe patterns, the landing slug must be configurable.

## Goal

Make `LANDING_SLUG` a first-class project profile variable that flows through to the landing-page module. Default to `""` (preserves the custom-domain primary-page behaviour post-v1.4.0). Non-empty values enable system-domain and redirect/iframe modes without further framework changes.

The framework deliberately does NOT model "hosting mode" as a concept — it just uses whatever `DOMAIN` and `LANDING_SLUG` the project config provides. Mode-awareness lives in the skill (separate plan), because different modes need different preflight checks and coaching. The framework is pure plumbing.

## Supported hosting modes (documented in `docs/framework.md`)

Same two values drive all modes; only the config differs:

| Mode | `DOMAIN` | `LANDING_SLUG` | HubSpot side | External side |
|------|----------|----------------|--------------|---------------|
| Custom-domain-primary | `heard.example.com` | `""` | Custom domain connected, `isPrimaryLandingPage = true` | CNAME → HubSpot CDN |
| System-domain | `147959629.hs-sites-eu1.com` | `"heard"` (non-empty) | No custom-domain work; system domain is auto-provisioned | Nothing |
| System-domain-redirect | `147959629.hs-sites-eu1.com` | `"heard"` | Same as system-domain | URL-forward `vanity.example.com` → `<system-domain>/<slug>` |
| System-domain-iframe (v1.7 roadmap) | `147959629.hs-sites-eu1.com` | `"heard"` | Same as system-domain | S3 + CloudFront with iframe wrapper at `vanity.example.com` |

The framework's code path is identical across all four. The skill's coaching and preflight differ (covered in the skill plan).

## Changes to `terraform/modules/landing-page/variables.tf`

Add:

```hcl
variable "landing_slug" {
  description = "Slug for the landing page URL. Empty string ('') serves the page at root of the connected primary-landing-page domain. A non-empty value (e.g. 'heard') places the page at <domain>/<slug>. See docs/framework.md for hosting-mode patterns."
  type        = string
  default     = ""
}

variable "thankyou_slug" {
  description = "Slug for the thank-you page URL."
  type        = string
  default     = "thank-you"
}
```

(`thankyou_slug` is likely already a variable; confirm and add if missing. Keeping them symmetric avoids surprise.)

## Changes to `terraform/modules/landing-page/pages.tf`

Replace the currently hard-coded `slug = ""` with `slug = var.landing_slug`:

```hcl
resource "restapi_object" "landing_page" {
  # ...
  data = jsonencode({
    name         = var.page_landing_name
    slug         = var.landing_slug
    domain       = var.domain
    htmlTitle    = var.page_landing_title
    templatePath = var.template_path_landing
    state        = "PUBLISHED"
    language     = "en-gb"
  })
}
```

Same pattern for `thankyou_page` resource — use `var.thankyou_slug`.

Note: v1.4.0 has `domain = ""`. This plan keeps that. Users who want a specific system-domain override set `DOMAIN` in their project profile to the system hostname; `var.domain` flows through unchanged.

Wait — there's a subtlety. In system-domain mode, `DOMAIN` is set to `<portal>.hs-sites-eu1.com`. That value must flow to `var.domain` in the module. Currently v1.4.0 makes pages.tf hard-code `domain = ""` (discarding `var.domain`). That needs to be **walked back to `domain = var.domain`** — let the project config decide.

Updated plan for pages.tf:

```hcl
data = jsonencode({
  name         = var.page_landing_name
  slug         = var.landing_slug
  domain       = var.domain      # empty string for custom-domain mode, system hostname for system-domain mode
  # ...
})
```

This is a behavioural reverse from v1.4.0's fix. Explanation for the change:

- v1.4.0 set `domain = ""` because TSC's working page had empty domain and non-UUID slug was generated by HubSpot in response to `""` on a non-connected domain. The fix assumed the custom-domain primary-page pattern.
- With this plan, `domain` becomes the user's explicit statement of which domain to publish to. For custom-domain mode, set `DOMAIN=""`. For system-domain mode, set `DOMAIN="<portal>.hs-sites-eu1.com"`. Both work because the user (via account/project profile) knows what they want.

Document this carefully: the account profile's `DOMAIN` variable takes different shapes per mode.

## Changes to `scaffold/terraform/main.tf`

Pass through the new variables to the module call:

```hcl
module "landing_page" {
  source = "git::https://github.com/digital-mercenaries-ltd/hs-lander//terraform/modules/landing-page?ref=v1.4.0"
  # ...
  landing_slug  = var.landing_slug
  thankyou_slug = var.thankyou_slug
  # ...
}

variable "landing_slug" {
  type    = string
  default = ""
}

variable "thankyou_slug" {
  type    = string
  default = "thank-you"
}
```

## Changes to `scripts/tf.sh`

Export new `TF_VAR_*` from the project profile:

```bash
export TF_VAR_landing_slug="${LANDING_SLUG:-}"
export TF_VAR_thankyou_slug="${THANKYOU_SLUG:-thank-you}"
```

## Changes to `scaffold/project.config.example.sh`

Add:

```bash
# Landing page URL path. Leave empty for custom-domain-primary mode (serves at root).
# Set to a slug like "heard" for system-domain or redirect/iframe modes.
LANDING_SLUG=""

# Thank-you page URL path.
THANKYOU_SLUG="thank-you"
```

## Changes to `~/.config/hs-lander/<account>/<project>.sh` schema

Add `LANDING_SLUG` as an optional field (default `""`). Empty means "custom-domain primary-page mode"; non-empty means "explicit slug".

`set-project-field.sh` needs to accept `LANDING_SLUG` and `THANKYOU_SLUG` as valid field keys (the script validates keys against a schema).

## Changes to `docs/framework.md`

New "Hosting modes" section with the table from the "Supported hosting modes" section above, explaining:

- How to configure each mode via `DOMAIN` and `LANDING_SLUG` values
- What HubSpot UI steps (if any) are needed per mode
- Public URL expectations per mode
- How to switch modes later (it's just a config change; no re-create of resources)

The pages.tf change from v1.4.0 — `domain = ""` — is partially walked back here. ## Verification

After implementation:

1. Existing custom-domain projects continue to work with no config change (default `LANDING_SLUG=""` preserves current behaviour)
2. A project with `DOMAIN="147959629.hs-sites-eu1.com"` and `LANDING_SLUG="heard"` produces a page URL of `https://147959629.hs-sites-eu1.com/heard` after apply
3. `npm run tf:plan` shows the expected `slug` and `domain` values in the rendered resource payload per mode
4. All existing tests pass; new hosting-mode assertions pass
5. `docs/framework.md` has the "Hosting modes" section with the decision table
