# Plan: v1.6.3 — Forms `privacyText` requirement

**Date:** 2026-04-23
**Status:** Pending
**Scope:** Framework. Add the `privacyText` field required by HubSpot Forms v3 when `legalConsentOptions.type = "implicit_consent_to_process"` is set.
**Target release:** v1.6.3 (patch — new optional module input with a sensible default; no breaking change)
**Dependency:** v1.6.2 shipped

## Problem

Heard's 2026-04-23 apply against v1.6.2 cleared the landing-page and welcome-email blockers. Both forms now error at the same point:

```
Some required fields were not set: [privacyText]
```

Error column pointer (549 for capture, 1093 for survey) puts the missing field adjacent to `legalConsentOptions`. Forms v3 now requires `privacyText` when `legalConsentOptions.type = "implicit_consent_to_process"` — previously accepted without it, now rejected.

The `legalConsentOptions.privacyText` field is the short disclosure shown beneath form fields explaining what the submitter's data will be used for. Required for GDPR-style consent flows, which is why it's gated on `implicit_consent_to_process`.

## Design

### Fix 1 — New module input `privacy_text`

`terraform/modules/landing-page/variables.tf` gains:

```hcl
variable "privacy_text" {
  type        = string
  default     = "We'll use the information you provide to send you occasional updates. You can unsubscribe at any time."
  description = "Short privacy disclosure shown beneath form fields. Required by HubSpot Forms v3 when legalConsentOptions.type = implicit_consent_to_process."
}
```

Default is a generic, GDPR-adequate disclosure that works as-is for most projects. Consumers with specific legal text override per-project.

### Fix 2 — Wire into both form payloads

`terraform/modules/landing-page/forms.tf`: both `capture_form` and `survey_form` gain `privacyText = var.privacy_text` inside `legalConsentOptions`:

```hcl
legalConsentOptions = {
  type        = "implicit_consent_to_process"
  privacyText = var.privacy_text
}
```

Header comment on `forms.tf` gets a line noting the v3 requirement so a future contributor doesn't delete it.

## Downstream consumers

- **Heard**: will pick up the default automatically. If the default prose doesn't fit, set `PRIVACY_TEXT` on the project profile (requires adding it to `set-project-field.sh` allow-list — see below) or override the module input directly in `terraform/main.tf`.
- **TSC**: pre-module era, unaffected.
- **Any v1.6.x consumer**: no action required; the default ships populated.

### Optional: `PRIVACY_TEXT` project-profile key

Low-priority; not strictly needed for Heard to unblock since the default is acceptable. If it's convenient during the same commit, add `PRIVACY_TEXT` to `scripts/set-project-field.sh`'s allow-list and to the project-profile stub in `scripts/scaffold-project.sh`. If not, defer — consumers can set the module input directly or override in the generated `main.tf`.

Recommendation: **defer** unless trivially easy. Each additional project-profile key expands the surface area the skill needs to reason about; a module default is sufficient for now.

## Scope limits

- No `PRIVACY_TEXT` project-profile key in this patch (see "Optional" above).
- No support for richer `legalConsentOptions` variants (`legitimate_interest`, `consent_checkboxes`). If/when needed, separate plan.
- No retroactive probe of existing forms on consumer portals — this is a create-time validation; consumers won't hit it until their next form-creating apply.
