# Plan: v1.6.5 — Email publish step + upload endpoint fix

**Date:** 2026-04-23
**Status:** Pending
**Scope:** Framework. Two source fixes surfaced when Heard re-applied against v1.6.4 and then deployed.
**Target release:** v1.6.5 (patch — one Terraform-resource addition + one one-line script fix)
**Dependency:** v1.6.4 shipped

## Problem

### Issue 1 — Welcome email POST rejects `state = "AUTOMATED"`

Heard's `terraform taint` + apply destroyed the stuck pre-v1.6.0 email (395263518907) and tried to POST a fresh one with v1.6.0's create payload (`state = "AUTOMATED"`, `isPublished = true`). HubSpot returned:

```
Creating an email in the published state AUTOMATED is not allowed.
Consider using the DRAFT state AUTOMATED_DRAFT.
```

HubSpot's Marketing Email API v3 enforces that automated emails are created as `AUTOMATED_DRAFT` and promoted to `AUTOMATED` via a separate `POST /marketing/v3/emails/{id}/publish` call. The v1.6.0 design (which made the v1.0.0 → v1.6.0 taint recovery possible) was based on probing an already-published email — we never tested the create path end-to-end because the test portal's email already existed.

Heard's portal now has **no welcome email at all** — the old one was destroyed, the new one didn't land. Signups would receive nothing until this is fixed.

### Issue 2 — Design Manager upload endpoint rejects `/developer/content`

`npm run deploy` returned 8/8 failed uploads (HTTP 415):

```
Environment specified in path 'developer' is invalid.
Must be one of: draft, published
```

`scripts/upload.sh:27` hard-codes `API_BASE=".../cms/v3/source-code/developer/content"`. The `developer` environment is not a valid source-code environment in CMS v3 — must be `draft` (stage first) or `published` (go-live immediately).

Heard's portal currently has pages registered with `templatePath = "heard/templates/landing-page.html"` etc., but those files don't exist in Design Manager. The pages serve HubSpot's "template not found" fallback until an upload succeeds.

## Design

### Fix 1 — Email create-as-draft + publish step

Two edits to `terraform/modules/landing-page/emails.tf`:

1. **Change the create payload.** `data` block currently sends `state = "AUTOMATED"` and `isPublished = true`. Change to `state = "AUTOMATED_DRAFT"` and `isPublished = false`. `update_data` is unchanged (still omits state/isPublished).
2. **Add a post-create publish step.** Introduce a `terraform_data` resource that fires after the email is created and POSTs to the publish endpoint:

   ```hcl
   resource "terraform_data" "publish_welcome_email" {
     triggers_replace = [restapi_object.welcome_email.id]

     provisioner "local-exec" {
       command = <<-EOT
         bash "$HS_LANDER_PROJECT_DIR/scripts/hs-curl.sh" POST "/marketing/v3/emails/${restapi_object.welcome_email.id}/publish"
       EOT
     }
   }
   ```

   - `triggers_replace` keyed on the email's ID means the publish fires on first create and again whenever the email resource is recreated (e.g. after a taint).
   - The publish endpoint is idempotent — calling it on an already-published email is a no-op, so this is safe on subsequent applies where `terraform_data` doesn't replace but the `local-exec` provisioner doesn't rerun anyway (provisioners only run on create, not update).
   - `HS_LANDER_PROJECT_DIR` is exported by `scripts/tf.sh` before it invokes terraform, so local-exec inherits it. `hs-curl.sh` reads the token from Keychain — no secret leakage into the terraform execution environment.

**Alternative considered** — a `null_resource` instead of `terraform_data`. Functionally identical here; `terraform_data` is the modern equivalent introduced in TF 1.4 and preferred for new code.

**Alternative considered** — skipping the publish automation and documenting "manually promote via UI." Rejected: every fresh deploy would leave the email unsent until someone clicks Publish in HubSpot UI. The friction defeats the automation premise.

**Alternative considered** — calling the publish endpoint directly from `restapi_object` by configuring a custom `create_path` or chaining a second `restapi_object`. Rejected: provider doesn't cleanly model "POST to /resource, then POST to /resource/{id}/publish" as a single resource; local-exec is clearer.

### Fix 2 — Upload endpoint

`scripts/upload.sh:27` — `/developer/content` → `/published/content`. One-line change. Header comment gains a note explaining the v3 environment values (`draft`, `published`) so no future contributor reverts to `developer` from an older HubSpot CLI reference.

**Why `published` and not `draft`:** `published` is the current framework's behaviour contract (`npm run deploy` uploads to live). A `draft` workflow could be useful (stage before go-live), but that's a separate feature — the immediate fix just restores the expected path. Deferred to a later enhancement if consumers ask for it.

## Downstream consumers

- **Heard** (blocked): re-applies under v1.6.5. Expected plan:
  - `welcome_email`: CREATE (not in state after the v1.6.4 destroy; fresh email created as AUTOMATED_DRAFT, then published to AUTOMATED by the new `terraform_data.publish_welcome_email` provisioner).

- **Consumers on pre-v1.6.0 with a stuck email**: same `terraform taint module.landing_page.restapi_object.welcome_email` recovery path as documented in v1.6.2, but it now works because the recreate goes AUTOMATED_DRAFT → publish → AUTOMATED cleanly.
- **Scripts-drift consumers**: any project whose local `scripts/upload.sh` was copied from an earlier framework version needs the file refreshed. `hs-lander-refresh-scripts` is a pending roadmap item (see Follow-ups); until then, manual copy.

## Scope limits

- **Landing-page temp slug / domain-primary issue.** NOT in this patch — it's a portal configuration problem (heard.digitalmercenaries.ai isn't set as primary landing-page domain in Settings → Content → Domains & URLs), not a framework bug. See Follow-ups.
- **draft vs published upload workflow.** Only `published` is restored. A two-environment `npm run deploy -- --draft` is a future feature.
- **Publish idempotency edge cases.** If the publish endpoint returns non-200 on an already-published email in some future API revision, the provisioner will hard-fail apply. Acceptable for now; add error-handling if it becomes an issue.
