# Plan: v1.9.1 — Operator ergonomics + test pyramid widening

**Date:** 2026-04-28
**Status:** Pending
**Scope:** Framework. Five small additive changes that close gaps surfaced across v1.7.x–v1.9.0 deploys. No module input contract changes; no breaking changes to existing scripts. Adds two new operator-facing scripts (`migrate-project.sh`, `projects-describe.sh`), a new `PREFLIGHT_VERSION_DRIFT` line, a scaffold-rooted plan test, and the `VERSION.compat` maintenance process.
**Target release:** v1.9.1 (patch — additive ergonomics + test coverage).
**Dependency:** v1.9.0 shipped (the lib helpers + decomposed preflight that this plan builds on are now on `main`).

## Problem

Five threads from the v1.7.x–v1.9.0 deploy and review trail point at the same maintenance gap: it's too easy for a project to drift from the framework, and the framework has no command-level handle to surface the drift, migrate it, or describe per-project state with structured output.

### 1 — Project framework-version drift goes undetected until apply

A project's `terraform/main.tf` pins a specific framework tag (e.g. `?ref=v1.6.0`). The user has a newer framework installed (e.g. `v1.9.0`), bumps `VERSION` on every release, and copies new scripts via `upgrade-project-scripts.sh` — but the Terraform `?ref=` stays at the pinned version. The next `terraform apply` runs against old module code; module-API changes (new required inputs, renamed variables) silently miss.

**Concrete recurrence:** v1.8.1's scaffold-pin defect — `scaffold/terraform/main.tf` was stuck at `?ref=v1.6.0` from v1.6.0 through v1.8.0. Every release between updated `VERSION` without updating the scaffold pin. The miss was invisible until a fresh scaffold was run on a new project. The same failure mode applies to existing projects: their `?ref=` stays at whatever version they were scaffolded against until the operator manually edits.

A preflight-time check that compares the project's `?ref=` against the installed framework `VERSION` would surface the drift up front, before apply fails on a missing module input.

### 2 — Migration between framework versions is manual and error-prone

Migrating a project from `?ref=v1.6.0` to `?ref=v1.9.0` means:

1. Bumping the `?ref=` tag in `terraform/main.tf`
2. Reading every CHANGELOG entry between to find new required module inputs
3. Adding any new variables to the project profile via `set-project-field.sh`
4. Adding any new module-input wiring to `terraform/main.tf` itself (e.g. `email_reply_to`, `email_preview_text`, `auto_publish_welcome_email`, `capture_post_submit_action_override` — all introduced post-v1.6.0)
5. Running `tf:init`, `tf:plan`, inspecting, applying

Step 4 is the most error-prone — the operator has to reason about which root variables to declare and which to wire through. v1.8.1 had to fix this for fresh scaffolds; existing projects still pay the manual cost on every upgrade.

### 3 — Project profile inspection lacks structured output

The skill (and the operator) sometimes needs to know what's currently in a project profile — `DOMAIN`, `LANDING_SLUG`, the `CAPTURE_FORM_ID` that `post-apply.sh` wrote, etc. Today the only path is `Read` on `~/.config/hs-lander/<account>/<project>.sh`. That's permitted by the skill's Operating Rule 3, but it's also asymmetric with `accounts-describe.sh` which already provides structured `ACCOUNT_<FIELD>=<value>` output.

Structured output enables three things the unstructured `Read` doesn't:
- The skill (or any consumer) can parse without writing custom regex.
- The framework owns the canonical project-profile schema; consumers don't reverse-engineer it.
- A schema change (e.g. a new field added to `ALLOWED_KEYS` in `set-project-field.sh`) automatically flows through to the inspection command.

### 4 — The Terraform plan test bypasses the scaffold root

`tests/test-terraform-plan.sh` exercises `tests/fixtures/terraform/main.tf` — a hand-written fixture that calls the modules directly. The actual scaffold root (`scaffold/terraform/main.tf`) is never plan-tested. v1.8.1's scaffold-pin defect (stuck `?ref=v1.6.0`) and Component 2's scaffold root-variable wiring (the four new vars `email_reply_to`, `email_preview_text`, `auto_publish_welcome_email`, `capture_post_submit_action_override`) were both invisible to CI because the test bypasses the actual file.

A scaffold-rooted plan test would have caught both before they shipped. The cost is small — copy the existing fixture's invocation pattern, point at the scaffold's `main.tf` instead.

### 5 — `VERSION.compat` maintenance is undocumented

The roadmap (line 686) references a `VERSION.compat` file the skill reads via `check-framework-compat.sh` to decide whether the installed framework is compatible with the skill's expectations. The file does not exist yet. There's no documented process for who updates it, when, or how it's kept in sync with actual breaking changes. Worst case once it lands: it silently drifts and the compat check becomes either uselessly permissive (skill works against incompatible frameworks) or spuriously strict (skill blocks against compatible ones).

v1.9.1 introduces the file, documents the bump rules, and adds a CI guard that flags PRs that touch the contract surface without bumping `VERSION.compat`.

## Design

Each fix is small and self-contained. Order is presented as the implementer should land them — earlier items inform later ones.

### Fix 1 — `PREFLIGHT_VERSION_DRIFT` check

**New file:** `scripts/preflight.d/05-version-drift.sh`

Runs early in the preflight cascade (numeric prefix `05` — between `00-tools-required.sh` and `10-project-pointer.sh`) so version-drift is reported even before the rest of the pipeline runs. Reads two values:

- The installed framework version: `cat $FRAMEWORK_INSTALL_DIR/VERSION` (or `unknown` when absent).
- The project's pinned `?ref=`: parse from `$PROJECT_DIR/terraform/main.tf`. The pin shape is stable across the framework's history (`?ref=v1.X.Y` in a `git::https://github.com/.../...` URL); use a small awk extractor.

Three states emit:

```
PREFLIGHT_VERSION_DRIFT=ok                              — installed == pinned
PREFLIGHT_VERSION_DRIFT=warn pinned=<X> installed=<Y>   — installed > pinned (the realistic drift case)
PREFLIGHT_VERSION_DRIFT=skipped <reason>                — terraform/main.tf missing, ?ref= unparseable, or PROJECT_DIR missing
```

Non-blocking — `warn` and `skipped` both exit 0. The skill (or an operator) reads the line and can run `migrate-project.sh` (Fix 2) to address the drift.

A separate `installed > pinned` vs `installed < pinned` distinction is worth making in the warn detail: the realistic case is "installed newer than pinned", but a developer running an older framework against a project pinned to a newer version is also possible and should be surfaced clearly so they upgrade their framework rather than try to migrate the project backward.

The 17-line preflight contract becomes 18-line. `tests/test-preflight.sh` extends with three scenarios:
- Project pinned exactly at installed version → `ok`
- Project pinned older → `warn pinned=<X> installed=<Y>`
- Project's `terraform/main.tf` missing the `?ref=` line → `skipped (?ref not found in terraform/main.tf)`

`tests/test-preflight-decomposition.sh` line-count assertion bumps from 17 → 18.

### Fix 2 — `scripts/migrate-project.sh`

**New file.** Sanctioned helper for moving a project from one framework version to another. Replaces the manual sequence (edit `?ref=`, read CHANGELOG, add new module-input wiring, etc.) with a structured command.

**Invocation:**

```bash
bash $FRAMEWORK_HOME/scripts/migrate-project.sh <project-dir> <target-version>
```

`<project-dir>` defaults to `$PWD` if omitted. `<target-version>` defaults to the installed framework's `VERSION`.

**Behaviour:**

1. Read current `?ref=` from `<project-dir>/terraform/main.tf`. Refuse to proceed if the file is missing or the `?ref=` is unparseable (`MIGRATE=error main-tf-missing|ref-unparseable <detail>`, exit 1).
2. Compare current and target. If equal, emit `MIGRATE=noop pinned=<X>` and exit 0.
3. Validate target is in CHANGELOG.md — refuse to migrate to a version we don't have a documented record for (`MIGRATE=error unknown-target <X>`, exit 1).
4. Surface what's between (any "Migration" sections in the CHANGELOG between current and target, plus any new module-input requirements identifiable by grepping `scaffold/terraform/main.tf` from each version's tag for variables not present in the older version's tag).
5. Print a structured plan: `MIGRATE_STEP=<n> <description>` lines for each diff (e.g. `MIGRATE_STEP=1 bump ?ref=v1.6.0 → ?ref=v1.9.0 in terraform/main.tf`, `MIGRATE_STEP=2 declare email_reply_to root variable`, `MIGRATE_STEP=3 wire email_reply_to into module.landing_page block`).
6. Without a `--apply` flag, exit 0 with `MIGRATE=plan-only <n> steps`. Operator inspects, then re-runs with `--apply`.
7. With `--apply`, execute each step against the project's files. Each step is atomic (tempfile + `mv`); if any step fails, abort the whole migration with `MIGRATE=error step-<n>-failed <detail>` and exit 1.
8. On success, emit `MIGRATE=ok pinned=<X> → <Y>` and exit 0.

Atomic-write pattern + EXIT trap mirrors `set-project-field.sh`'s convention. Migration data lives in a new file `scripts/lib/migration-rules.sh` — a sourced bash file with a function per known migration (`migrate_v1_6_0_to_v1_7_0`, `migrate_v1_7_0_to_v1_8_0`, etc.). Each function knows what to add / rename / wire for that one-version step. Migrations chain: a v1.6.0 → v1.9.0 migration applies the four sequential one-version migrations.

Initial migrations to ship in v1.9.1's `migration-rules.sh`:
- v1.6.0 → v1.7.0: declare `email_preview_text`, `auto_publish_welcome_email`, `capture_post_submit_action_override` root variables; wire them into module.landing_page.
- v1.7.0 → v1.8.0: no scaffold-root changes (the module additions were optional).
- v1.8.0 → v1.8.1: declare `email_reply_to` root variable; wire it into module.landing_page (replacing the placeholder string from earlier scaffolds).
- v1.8.1 → v1.9.0: no scaffold-root changes (Component 1 added `npm run plan` + `npm run apply` to package.json — that's a separate file the script also touches).

Each migration step records its target line numbers / variable names in the rule definition; the script applies them via the lib helpers (`sed_inplace`, `sed_escape_replacement`).

Tests: `tests/test-migrate-project.sh` (new). Scenarios:
- Migrate from v1.6.0 fixture to v1.9.0: assert `?ref=v1.9.0` in result, all four new variables declared.
- Migrate to current installed version (default target): same.
- Migrate target equals current → noop, exit 0.
- Migrate target unknown to CHANGELOG → error, exit 1.
- `--plan-only` (default): emit MIGRATE_STEP lines without writing the file.
- Aborted migration leaves all files unchanged (atomic-write guarantee — `kill -INT` mid-run, assert tempfile gone and live file byte-identical).

### Fix 3 — `scripts/projects-describe.sh`

**New file.** Structured-output equivalent of `Read`-ing the project profile.

**Invocation:**

```bash
bash $FRAMEWORK_HOME/scripts/projects-describe.sh <account> <project>
```

**Output (when profile exists, exit 0):**

```
PROJECT_SLUG=<value>
PROJECT_DOMAIN=<value>
PROJECT_DM_UPLOAD_PATH=<value>
PROJECT_GA4_MEASUREMENT_ID=<value>
PROJECT_CAPTURE_FORM_ID=<value>
PROJECT_SURVEY_FORM_ID=<value>
PROJECT_LIST_ID=<value>
PROJECT_LANDING_SLUG=<value>
PROJECT_THANKYOU_SLUG=<value>
PROJECT_HUBSPOT_SUBSCRIPTION_ID=<value>
PROJECT_HUBSPOT_OFFICE_LOCATION_ID=<value>
PROJECT_EMAIL_PREVIEW_TEXT=<value>
PROJECT_AUTO_PUBLISH_WELCOME_EMAIL=<value>
PROJECT_EMAIL_REPLY_TO=<value>
```

**Output (profile missing, exit 1):**

```
PROJECT_STATUS=missing <path>
```

**Output (account name invalid via the v1.8.1 validate-name lib, exit 1):**

```
PROJECT_STATUS=error invalid-account-name '<name>' (...)
PROJECT_STATUS=error invalid-project-name '<name>' (...)
```

**Implementation:**

1. Source `scripts/lib/validate-name.sh`. Validate `<account>` and `<project>` arguments early.
2. Source `scripts/lib/source-vars.sh`. Use `source_vars` to extract every field in `set-project-field.sh`'s `ALLOWED_KEYS` array — single source of truth for the schema.
3. Print one `PROJECT_<KEY>=<value>` line per allowed key, including empty values (consistent with `accounts-describe.sh`'s pattern).

The structure mirrors `accounts-describe.sh` exactly — same `<TYPE>_<FIELD>=<value>` shape, same exit codes, same error prefixes. A skill that consumes `accounts-describe.sh` already has the parser logic; `projects-describe.sh` is identical with `PROJECT_` substituted for `ACCOUNT_`.

**Credential safety:** the script only reads project-profile fields, none of which carry credentials. The Keychain reference (`HUBSPOT_TOKEN_KEYCHAIN_SERVICE`) lives in account profiles and is intentionally not in `ALLOWED_KEYS` for project profiles.

Tests: `tests/test-projects-describe.sh` (new). Scenarios mirror `tests/test-accounts-describe.sh`:
- Complete profile → all 14 lines emitted with values
- Missing profile → exit 1 with `PROJECT_STATUS=missing <path>`
- Invalid name → exit 1 with `PROJECT_STATUS=error invalid-...`

### Fix 4 — Scaffold-rooted plan test

**New file:** `tests/test-scaffold-terraform-plan.sh`

Mirrors `tests/test-terraform-plan.sh` but exercises `scaffold/terraform/main.tf` instead of `tests/fixtures/terraform/main.tf`. Setup:

1. Create temp project directory.
2. Use `scaffold-project.sh` (or copy `scaffold/terraform/main.tf` directly) into the temp dir.
3. Provide a minimal `terraform.tfvars` (or env-var equivalents) for the required root variables.
4. Run `terraform init -backend=false` against the temp dir.
5. Run `terraform plan -out=...` and parse output via `terraform show -json`.

**Assertions:**

- The four post-v1.6.0 root variables (`email_reply_to`, `email_preview_text`, `auto_publish_welcome_email`, `capture_post_submit_action_override`) are declared in scaffold's `main.tf` and the plan shows them flowing through to `module.landing_page`. (Catches v1.8.1's wiring miss.)
- The `?ref=` pin matches the framework's current `VERSION`. (Catches v1.8.1's stuck `?ref=v1.6.0` defect.)
- The plan completes without errors against a clean state.
- The plan produces the same resource graph as `tests/test-terraform-plan.sh`'s fixture (same module instantiations, same `depends_on` edges).

The test uses the same restapi-provider mock pattern as `tests/test-terraform-plan.sh`; only the source `main.tf` differs.

This test runs in CI under a new job (or appended to the existing `plan-test` job — design TBD; one job is simpler unless the test footprint grows).

### Fix 5 — `VERSION.compat` maintenance process

**New file:** `VERSION.compat` at the repo root.

**Format:** single line, semantic version range expression. Initial value:

```
>=1.7.0,<2.0.0
```

The constraint declares "framework versions in this range are skill-compatible". The skill reads it via its `check-framework-compat.sh` against the project's installed `VERSION`.

**Bump rules** (documented in `CLAUDE.md` and a new `docs/version-compat.md`):

A `VERSION.compat` bump is required when any of the following change:
- A `PREFLIGHT_*` line's name, presence, or output state values
- A pre-scaffold or config-mutation script's prefix or state values (`SCAFFOLD=`, `INIT_POINTER=`, etc.)
- A module input variable's name or required-vs-optional status
- The build-token list in `build.sh`
- A scaffold root variable's name or default

Any of these is a change to the surface the skill talks to. `VERSION.compat`'s upper bound MAY tighten in a patch release if a contract change inadvertently broke compatibility (rare); the lower bound moves forward with each new minimum-required-by-skill feature.

**CI guard:** new `.github/workflows/ci.yml` job `version-compat-check` that:

1. Runs on every PR.
2. Diffs the changed files vs `main`.
3. If any of the contract-surface files (listed above — preflight, scripts, scaffold root, build.sh) changed AND `VERSION.compat` did NOT change in the same PR, fail with: "Contract surface changed but VERSION.compat not bumped. Either bump it (touching the contract surface implies a skill-facing change) or document why this is internal-only in the PR description."

The check is a soft guard — operator can override by adding a `[skip-compat-check]` token to the PR description. Document in the PR template (when one is written).

Tests: not strictly needed for `VERSION.compat` itself (it's a one-line file), but a new fixture in `tests/test-version.sh` confirms the file exists and matches the documented format.

### Coordinated changes

All five fixes touch related areas; landing them in a single PR (or three small PRs at most) keeps the cost low:

- **PR A:** Fix 1 (`PREFLIGHT_VERSION_DRIFT`) + Fix 4 (scaffold-rooted plan test). These exercise the contract through preflight and CI; both close v1.8.1's scaffold-pin defect from different angles.
- **PR B:** Fix 2 (`migrate-project.sh`) + Fix 3 (`projects-describe.sh`). Operator-facing scripts; share lib usage and test patterns.
- **PR C:** Fix 5 (`VERSION.compat` + CI guard). Standalone; can land before, between, or after the others.

Single-PR alternative: bundle as one v1.9.1 PR if review surface stays under ~600 lines net. Probably feasible.

## Follow-ups (not in this plan)

- **Skill v1.9.1 adoption:** flag for the next skill-session to consume the new commands. Specifically the skill should:
  - Call `projects-describe.sh` instead of `Read` for project-profile lookups.
  - Read `PREFLIGHT_VERSION_DRIFT` and coach the user toward `migrate-project.sh` when it warns.
  - Use `migrate-project.sh --plan-only` to surface the migration plan to the user before invoking with `--apply`.

- **`migrate-project.sh` migration rules** as new releases ship: each release that adds a scaffold root variable, renames an existing one, or changes a project-profile schema field must register a migration rule in `scripts/lib/migration-rules.sh`. Document in the release-checklist (probably in `CONTRIBUTING.md` once written).

- **`VERSION.compat` granularity:** the initial constraint is coarse (`>=1.7.0,<2.0.0`). If a v1.x release breaks skill compatibility within the major (rare but possible), tighten to a narrower upper bound. The CI guard catches the trigger; the response is human judgement.

- **`projects-describe.sh` field schema split:** as new fields land in `set-project-field.sh`'s `ALLOWED_KEYS`, the describe command picks them up automatically (it reads the same array). The opposite direction — adding a field via `projects-describe.sh` first then `set-project-field.sh` — is not supported; the schema's source of truth is the allow-list.
