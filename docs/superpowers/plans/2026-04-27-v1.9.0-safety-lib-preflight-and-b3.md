## Why bundle these five

Patch-vs-minor distinction matters for tag discipline and CHANGELOG legibility, but more importantly the five parts share infrastructure:

- The lib consolidation absorbs the safety pair's `backup-file.sh` and the B3 work's `local-exec` helpers. Landing them separately means writing the same sed/keychain helpers twice.
- The preflight decomposition exposes the existing internal helpers (`_source_vars`, `_extract_var`, `_sed_inplace`) to the rest of the codebase as lib functions. The carryover items (PREFLIGHT_EMAIL_REPLY_TO observability, validate-name unit tests) become trivial once the lib is in place.
- B3 needs `terraform_data` orchestration that wants the same backup-before-apply hook the safety pair installs.
- The carryovers each take one or two lines once the lib helpers exist; spreading them across four PRs is more ceremony than the changes warrant.

A single v1.9.0 release with one cohesive CHANGELOG entry serves consumers better than five back-to-back v1.8.x releases.

## Component 2 — `scripts/lib/` consolidation

The framework currently has `scripts/lib/tier-classify.sh` (v1.7.0) and `scripts/lib/validate-name.sh` (v1.8.1). Several patterns that should live in lib are inlined across multiple scripts. Consolidating them now:

1. Reduces the surface for divergence (the `_sed_escape` shape is duplicated in `build.sh` and `set-project-field.sh` today; if one drifts, build silently corrupts).
2. Makes the carryover defensive items trivial (e.g. `${1:-}` guard in `is_valid_name` is one line in lib, not five copy-edits).
3. Provides the foundation v1.9.1's R9 ergonomics work depends on (`migrate-project.sh`, `projects-describe.sh`).

### New lib files

**`scripts/lib/sed-portable.sh`** — extracted from the eight call sites that currently inline `_sed_inplace`:

```bash
# Portable in-place sed (BSD on macOS, GNU on Linux).
sed_inplace() {
  if [[ "$(uname)" == "Darwin" ]]; then
    sed -i '' "$@"
  else
    sed -i "$@"
  fi
}

# Escape sed replacement-side metacharacters: |, &, \. Replacement-side
# only — left-hand-side escaping is the caller's concern.
sed_escape_replacement() {
  printf '%s' "$1" | sed -e 's/[\\|&]/\\&/g'
}
```

Sourcing call sites: `build.sh`, `post-apply.sh`, `set-project-field.sh`, `accounts-init.sh`, `upgrade-project-scripts.sh`, plus future preflight check files (Component 3).

Carryover landing: closes silent-failure-hunter's H2 ("`_sed_escape` extracted to lib").

**`scripts/lib/source-vars.sh`** — extracted from `preflight.sh`'s `_source_vars` and `init-project-pointer.sh`'s `_extract_var`:

```bash
# Extract one or more KEY values from a shell-syntax file without sourcing.
# Emits canonical `KEY="value"` lines that callers can `eval`.
# Handles: KEY="value" | KEY='value' | KEY=value | export KEY=...
source_vars() {
  local path="$1"; shift
  local var
  for var in "$@"; do
    awk -v V="$var" '
      match($0, "^[[:space:]]*(export[[:space:]]+)?"V"=") {
        rest = substr($0, RSTART + RLENGTH)
        if (rest ~ /^"/) { if (match(rest, /^"[^"]*"/)) { printf "%s=%s\n", V, substr(rest, 1, RLENGTH); exit } }
        else if (rest ~ /^\x27/) { if (match(rest, /^\x27[^\x27]*\x27/)) { printf "%s=%s\n", V, substr(rest, 1, RLENGTH); exit } }
        else { if (match(rest, /^[^[:space:]#]+/)) { printf "%s=\"%s\"\n", V, substr(rest, 1, RLENGTH); exit } }
      }
    ' "$path"
  done
}
```

Sourcing call sites: `preflight.sh` (replaces inline `_source_vars`), `init-project-pointer.sh` (replaces inline `_extract_var`), `post-apply.sh` (used by the migration note in Component 5 carryover for the `EMAIL_REPLY_TO` observability check).

**`scripts/lib/keychain.sh`** — extracts the macOS Keychain read with xtrace suppression baked in. Currently inlined in `tf.sh`, `hs-curl.sh`, `upload.sh`, `preflight.sh`. Inconsistent xtrace handling across them is a real security gap (the v1.8.1 plan flagged "xtrace suppression in tf.sh / upload.sh / hs-curl.sh" as deferred to v1.9.0):

```bash
# Read a Keychain entry by service name, with bash xtrace temporarily
# suppressed so `bash -x scripts/<caller>.sh` does not leak the token.
# Returns 0 on success (token on stdout); non-zero with a structured
# error message on stderr otherwise.
keychain_read() {
  local service="$1"
  local was_xtrace=0
  case "$-" in *x*) was_xtrace=1; set +x ;; esac

  local token rc
  if token=$(security find-generic-password -s "$service" -a "$USER" -w 2>/dev/null); then
    rc=0
  else
    rc=$?
    echo "ERROR: Could not read Keychain entry '$service' (security exit $rc). Add it with:" >&2
    echo "  security add-generic-password -s '$service' -a \"\$USER\" -w 'TOKEN'" >&2
  fi

  if [[ "$rc" == 0 ]]; then
    printf '%s' "$token"
  fi

  (( was_xtrace == 1 )) && set -x
  return "$rc"
}
```

Sourcing call sites: `tf.sh`, `hs-curl.sh`, `upload.sh`, `preflight.sh`. All four currently re-implement the same `security find-generic-password` call; only `preflight.sh` has xtrace suppression today.

**`scripts/lib/validate-name.sh`** — already exists (v1.8.1 seed). Extend with the carryover defensive items from the v1.8.1 review:

```bash
is_valid_name() {
  # Defensive ${1:-} so callers under `set -u` who pass an unset var
  # don't crash with "$1: unbound variable" before validation runs.
  if (( $# != 1 )); then
    echo "is_valid_name: expected 1 arg, got $#" >&2
    return 2
  fi
  [[ "${1:-}" =~ ^[a-z0-9][a-z0-9-]*$ ]]
}
```

Carryover landing: closes silent-failure-hunter's H3 (`${1:-}` guard) and H4 (arg-count guard).

**Switch `accounts-init.sh` from inline regex to `is_valid_name`** — the v1.8.1 plan deferred this on the grounds that the lib was new. With v1.9.0's broader lib adoption, the switch is consistent and removes the last divergent-regex risk.

## Component 3 — `preflight.sh` decomposition

`scripts/preflight.sh` is now 727+ lines. Each release adds 50–80 lines (a new `PREFLIGHT_*` check + its dependencies). The maintainer concern (>700 lines) has been triggered.

### Approach: one check per file, thin runner

```
scripts/preflight.sh                # thin runner — sources lib, sources each check, emits the contract
scripts/preflight.d/
  00-tools-required.sh              # PREFLIGHT_TOOLS_REQUIRED
  10-project-pointer.sh             # PREFLIGHT_PROJECT_POINTER
  20-account-profile.sh             # PREFLIGHT_ACCOUNT_PROFILE
  30-project-profile.sh             # PREFLIGHT_PROJECT_PROFILE
  40-credential.sh                  # PREFLIGHT_CREDENTIAL
  50-api-access.sh                  # PREFLIGHT_API_ACCESS
  55-tier.sh                        # PREFLIGHT_TIER
  60-scopes.sh                      # PREFLIGHT_SCOPES
  65-project-source.sh              # PREFLIGHT_PROJECT_SOURCE
  70-dns.sh                         # PREFLIGHT_DNS
  75-domain-connected.sh            # PREFLIGHT_DOMAIN_CONNECTED
  80-email-dns.sh                   # PREFLIGHT_EMAIL_DNS
  85-email-reply-to.sh              # PREFLIGHT_EMAIL_REPLY_TO  (new — see Component 5)
  90-ga4.sh                         # PREFLIGHT_GA4
  95-form-ids.sh                    # PREFLIGHT_FORM_IDS
  99-tools-optional.sh              # PREFLIGHT_TOOLS_OPTIONAL
```

Numeric prefixes drive ordering. The runner globs `preflight.d/*.sh`, sorts, sources each in turn. Each file is responsible for emitting exactly one `PREFLIGHT_<NAME>=<state> [detail]` line.

### Shared state between checks

Some checks are no-ops or report `skipped` based on earlier check results (e.g. CREDENTIAL is `skipped` when ACCOUNT_PROFILE is missing). Today this is enforced by inline `if account_profile_ok == 1` guards. After decomposition:

- The runner exposes a small set of shared variables documented in `preflight.d/README.md`: `tools_required_ok`, `project_pointer_ok`, `account_profile_ok`, `project_profile_ok`, `credential_ok`, `api_access_ok`, `tier`. Each check file reads the ones it depends on and writes the one it owns.
- Documenting the dependency graph lets a maintainer add a new check (e.g. for a future `PREFLIGHT_<X>` line) without re-reading the whole script.

### Output contract preserved verbatim

The 16-line `PREFLIGHT_*` contract (15 with `EMAIL_REPLY_TO` adding to 17 — see Component 5) is invariant. `tests/test-preflight.sh` is the canary: the existing 174 assertions pass against the decomposed version with zero edits.

A new test (`tests/test-preflight-decomposition.sh`) asserts that:
- Removing one check file from `preflight.d/` causes the corresponding `PREFLIGHT_*` line to be missing from output (line-count drops by exactly 1).
- The runner refuses to run if `preflight.d/` is empty or missing.
- File ordering is stable across runs (sort on numeric prefix).

### Migration

`preflight.sh` itself shrinks from 727 lines to ~80 (the runner). Each check file is 30–80 lines. Total LOC goes up slightly (per-file boilerplate: shebang, source lib, brief header) but the maintainer-facing surface — "I'm adding a new check, where do I edit?" — collapses from "scroll through 727 lines of intermingled check logic" to "create a new file under preflight.d/, copy the template".

`scaffold-project.sh` already copies `scripts/lib/`; it now also needs to copy `scripts/preflight.d/`. One-line addition to the existing copy loop.

### 5.4 — `PREFLIGHT_EMAIL_REPLY_TO` observability line (silent-failure M2)

**New check.** Currently when `EMAIL_REPLY_TO` is empty in the project profile, `PREFLIGHT_EMAIL_DNS` silently falls back to probing `DOMAIN`. Operators have no way to tell from preflight output whether the explicit reply-to is in use. Symptom: a typoed `EMAIL_REPLY_TO=hello.example.com` (missing `@`) gets rejected by `set-project-field.sh`'s allow-list — but a forgotten `EMAIL_REPLY_TO` is silently empty, and DKIM probes against the wrong domain pass or fail without surfacing the cause.

**New `preflight.d/85-email-reply-to.sh`** emits one of:

- `PREFLIGHT_EMAIL_REPLY_TO=set <address>` — explicit value present in the profile
- `PREFLIGHT_EMAIL_REPLY_TO=fallback <DOMAIN>` — empty in profile; falling back to `DOMAIN`
- `PREFLIGHT_EMAIL_REPLY_TO=skipped` — when project profile is missing or incomplete (matches the existing skip-cascade pattern)

Non-blocking — both `set` and `fallback` exit 0. The line is informational; the skill or operator decides whether the fallback is acceptable for their setup.

The 16-line preflight contract becomes 17-line. `tests/test-preflight.sh` extends the line-count assertion accordingly and adds two scenarios: explicit set and fallback.

### 5.5 — Property type allow-list validation (silent-failure M4)

`terraform/modules/landing-page/variables.tf` adds a `validation` block to `custom_properties`:

```hcl
variable "custom_properties" {
  type = list(object({
    name      = string
    label     = string
    type      = string
    fieldType = optional(string, "")
    groupName = optional(string, "contactinformation")
    options   = optional(list(string), [])
  }))
  default = []

  validation {
    condition = alltrue([
      for p in var.custom_properties :
      contains(["string", "number", "enumeration", "bool"], p.type)
    ])
    error_message = "custom_properties[*].type must be one of: string, number, enumeration, bool. Other values are rejected by HubSpot's CRM API."
  }
}
```

Same shape for `survey_fields` (already partially validated; tighten the existing block to enumerate all four supported types).

Plan-time error replaces a runtime HubSpot 400 — the failure surface moves from "after `terraform apply` runs partway" to "at `terraform plan`, before any API call". Significantly better operator ergonomics on a bad config.

`tests/test-terraform-plan.sh` adds a fixture variant with `custom_properties = [{name=..., type="datetime", ...}]` (an unsupported type) and asserts `terraform plan` fails with the validation error. This requires a separate fixture directory (the main fixture must continue to plan cleanly); the test runs both and asserts the bad-fixture's plan failure mode.

### 5.6 — `_sed_escape` `inherit_errexit` defensive (silent-failure M1)

`build.sh` (and any other script that uses `$(...)` with `_sed_escape` under `set -e`) needs `shopt -s inherit_errexit` so a sed crash inside the command substitution propagates instead of silently producing an empty value.

One-line addition near the top of `build.sh` (and other scripts where the same pattern applies). Defensive — sed crashing in this code path is near-zero probability — but cheap and the cost of the alternative (silent corruption when the assertion is false) is high.

### 5.7 — Comment rot cleanup (comment-analyzer)

Strip Heard / v1.8.0 / v1.8.1 attribution from three terraform comments in `forms.tf` and `properties.tf`. Keep the technical rationale; move provenance to changelog/git blame. One-pass cleanup, ~10 lines deleted.

The replacement comments preserve the load-bearing technical content (parallelisation race, options requirement, fieldType vocabulary) and drop the "X's deploy hit this" framing that will be opaque to maintainers in 12 months.

### 5.8 — `_update_field` non-atomic write (silent-failure L2)

`post-apply.sh::_update_field` writes directly to the live profile. If a SIGINT interrupts between the three field updates, the profile is partially rewritten. `set-project-field.sh` already does the right thing (tempfile + atomic mv with EXIT trap); align `_update_field` to the same pattern.

Refactor `post-apply.sh` to: copy profile to tempfile, run all three `_update_field` calls against the tempfile, atomic-`mv` to live path on success, EXIT trap removes the tempfile.

This change is small (~10 lines) but earns its place by closing the partial-write recovery hole the v1.8.1 hardening surfaced.

## Implementation sequence

Component dependencies allow sequential implementation in this order. Each step is a self-contained PR; v1.9.0 ships when all six merge:

1. **Lib consolidation (Component 2).** Land first because Components 3 (preflight decomposition) and 5 (carryovers) reference lib helpers. PR scope: new lib files, switch existing inlined call sites, unit tests for each lib. Adoption order: validate-name extension → sed-portable → source-vars → keychain. This sequence keeps PR size bounded; the keychain extraction is the largest because it touches four scripts.

2. **`preflight.sh` decomposition (Component 3).** Depends on lib helpers landing first (the new check files source `scripts/lib/source-vars.sh` and similar). PR scope: thin runner, 16 check files under `preflight.d/`, new decomposition test, scaffold copy-loop update. Existing `tests/test-preflight.sh` runs unchanged as the regression canary.

3. **Carryover items (Component 5).** Most ride Component 2's lib changes; the remainder (PREFLIGHT_EMAIL_REPLY_TO, type-validation, comment cleanup, atomic write) are small standalone changes. Bundle into one PR with the lib-extension follow-ups (5.1, 5.2, 5.3 are absorbed; 5.4, 5.5, 5.6, 5.7, 5.8 are explicit additions).

4. **Safety pair (Component 1).** Independent of 1–3. Two PRs (one per component plan); land in either order. Coordinates with Component 4 only at apply time (state backup happens before the new B3 orchestration runs — natural fit).

5. **B3 probes (Component 4 prerequisites).** Run probes against a live portal (Heard's or a dedicated test portal). Capture findings in `references/hubspot-api-quirks.md`. PR scope: docs only; expand the B3 stub plan with prescriptive sections.

6. **B3 implementation (Component 4).** Depends on probe findings (step 5). Scope: `terraform/modules/landing-page/emails.tf` rewrite, new fixture, new tests. If the probes contradict the stub's recommended Option A, the implementation slips to v1.9.1 and v1.9.0 ships steps 1–5 + a B3 status update.
