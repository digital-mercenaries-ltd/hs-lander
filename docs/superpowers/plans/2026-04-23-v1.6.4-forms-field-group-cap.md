# Plan: v1.6.4 — Forms fieldGroup 3-field cap

**Date:** 2026-04-23
**Status:** Pending
**Scope:** Framework. Split form fields across multiple `fieldGroups` so no group exceeds HubSpot Forms v3's 3-field cap.
**Target release:** v1.6.4 (patch — internal payload restructure; no module input changes)
**Dependency:** v1.6.3 shipped

## Problem

Heard's 2026-04-23 apply against v1.6.3 landed `capture_form` (2 fields: email + project_source — under the cap) but `survey_form` errored:

```
There are too many fields in this group. A group should be no more than 3 fields.
FormFieldError.FIELD_GROUP_TOO_MANY_FIELDS
```

`terraform/modules/landing-page/forms.tf` currently emits a single `fieldGroup` per form, concatenating all fields:

```hcl
fields = concat(
  [email_field],                       # 1
  [for f in var.survey_fields: {...}], # N
  [project_source_field],              # 1
)
```

HubSpot Forms v3 enforces a hard cap of 3 fields per `fieldGroup` at validation time. Heard's survey has 4 survey questions (email + 4 + project_source = 6 fields in one group) and tripped the cap. The capture form passes today because Heard's `capture_form_fields` is empty, but any consumer with ≥2 capture fields hits the same error.

## Design

Replace the single-group payload with a list of groups, each holding ≤3 fields. Apply to both `capture_form` and `survey_form`.

### Partitioning scheme

Three groups, deterministically constructed:

1. **Email group** — always `{ groupType, richTextType = "text", fields = [email_field] }`. Single field; leaves the gateway input visually separated from subsequent questions.
2. **User-field groups** — `var.capture_form_fields` (or `var.survey_fields`) chunked into slices of 3, one group per slice. If the input list is empty, no user-field groups emit.
3. **Segmentation group** — always `{ groupType, richTextType = "text", fields = [project_source_field] }`. Hidden by scaffold CSS regardless of group placement.

Example — Heard survey with 4 fields:

```
Group 1: [email]
Group 2: [survey_q1, survey_q2, survey_q3]
Group 3: [survey_q4]
Group 4: [project_source]
```

### Why fixed partitioning, not caller-specified grouping

The alternative — changing `capture_form_fields` / `survey_fields` from `list(object)` to `list(list(object))` so consumers choose their own groups — is a breaking module-contract change. Fixed partitioning is non-breaking and covers every current use case. If a consumer ever needs control over group boundaries (for rich-text dividers between sections, say), that's a future minor-version enhancement with its own plan.

### Helper local

Terraform's `chunklist(list, size)` function is built-in and produces the exact slicing we need. No custom helper required:

```hcl
locals {
  survey_field_chunks = chunklist(var.survey_fields, 3)
}
```

`chunklist([], 3)` returns `[]`, so the no-fields-provided path is clean.

### Generated `fieldGroups` shape

```hcl
fieldGroups = concat(
  # 1. Email group
  [
    {
      groupType    = "default_group"
      richTextType = "text"
      fields = [
        { name = "email", label = "Email", fieldType = "email", ... }
      ]
    }
  ],
  # 2. User-field groups
  [
    for chunk in chunklist(var.survey_fields, 3) : {
      groupType    = "default_group"
      richTextType = "text"
      fields = [
        for field in chunk : {
          name         = field.name
          label        = field.label
          fieldType    = field.type
          objectTypeId = "0-1"
          required     = field.required
        }
      ]
    }
  ],
  # 3. Segmentation group
  [
    {
      groupType    = "default_group"
      richTextType = "text"
      fields = [
        {
          name         = "project_source"
          label        = "Project Source"
          fieldType    = "single_line_text"
          objectTypeId = "0-1"
          defaultValue = var.project_slug
        }
      ]
    }
  ]
)
```

Same pattern for `capture_form`, substituting `var.capture_form_fields` in the middle.

Header comment on `forms.tf` gets a line explaining the cap so future contributors don't collapse the groups.

## Scope limits

- No caller-controlled grouping (breaking change; deferred).
- No visual-separator injection between groups (rich-text dividers could live between groups in a future enhancement; out of scope here).
- Rendered form aesthetics: the multi-group form may visually stack differently than a single-group one depending on HubSpot's CSS. If a consumer wants all fields in a visually continuous flow they control via custom CSS; framework concern is just "apply succeeds."
