# Plan: v1.6.1 — Mastercard/restapi provider version bump

**Date:** 2026-04-22
**Status:** Pending
**Scope:** Framework. Bump the Mastercard/restapi provider version constraint from `~> 1.19` to `~> 2.0` so modules can use attributes introduced in the provider's 2.0.0 release.
**Target release:** v1.6.1 (patch — constraint bump only; no module contract change)
**Dependency:** v1.6.0 tagged (done)

## Problem

v1.6.0's `terraform/modules/landing-page/lists.tf` uses `ignore_all_server_changes = true` to suppress drift detection on HubSpot's wrapped list response. That attribute was added to the Mastercard/restapi provider in **v2.0.0 (April 2024)**.

Both framework modules still declare `required_providers { restapi { version = "~> 1.19" } }`:

- `terraform/modules/landing-page/main.tf:13`
- `terraform/modules/account-setup/main.tf:12`

Terraform intersects module and root-level provider constraints. With the modules requiring `~> 1.19`, the highest installable provider version is 1.20.0 — which does not expose `ignore_all_server_changes`. An apply using the v1.6.0 modules fails at plan time with an unknown-attribute error.

Heard's 2026-04-22 re-deploy session confirmed this: bumping the root project's constraint to `~> 2.0` produced `no suitable version is available` because `~> 1.19, ~> 2.0` has no solution. The fix must live in the framework modules, not the consuming project.

## Design

Bump the constraint from `~> 1.19` to `~> 2.0` in three files:

- `scaffold/terraform/main.tf`
- `terraform/modules/landing-page/main.tf`
- `terraform/modules/account-setup/main.tf`

### Why `~> 2.0` and not a wider range

The `~> 2.0` constraint caps at 2.x. The Mastercard/restapi 2.0.0 changelog flags 2.x as additive over 1.x (new attributes like `ignore_changes_to`, `ignore_all_server_changes`, `ignore_server_additions`, `read_search.search_patch`). Existing 1.x state migrates to 2.x without replacement. v3.0.0 is a plugin-framework rewrite and will require a dedicated migration plan — not included here. Capping at 2.x avoids surprise upgrades.

### Why a patch release (v1.6.1) and not v2.0.0 of the framework

The module contract (inputs, outputs, preflight output lines, scripts) is unchanged. Consumers pinning `?ref=v1.6.0` hit a provider-availability bug; `?ref=v1.6.1` resolves it with no edits to their `terraform/main.tf` beyond bumping the ref. Semver: bug fix — patch.

### Downstream consumers

- **Heard**: currently pinned to `?ref=v1.6.0`, provider at `~> 1.19`. After v1.6.1 is tagged, Heard bumps the ref and re-runs `tf:init -upgrade`. Terraform will select a 2.x provider automatically. State migrates cleanly because Heard has no `restapi_object` resources committed to state yet (the partial apply was rolled back in the previous session).
- **TSC**: pinned to framework v1.0.0 via direct file copy (pre-module era). Unaffected.
- **Any consumer already on ~> 1.19 with committed 1.x provider state**: terraform will upgrade the provider on next `init -upgrade`. No resource drift expected — the 2.x attributes the modules use (`ignore_all_server_changes`) are opt-in; the existing resource payloads are compatible.

## Scripts and tests

No script changes. No test updates required — the framework's test suite doesn't exercise provider-version compatibility directly.
