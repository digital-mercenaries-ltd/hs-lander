# Plan: v1.7.0 — HubL scaffold, tier-aware preflight, email anatomy

**Date:** 2026-04-26
**Status:** Complete (shipped as v1.7.0 on 2026-04-26)
**Scope:** Framework. Scaffold rewrite (templates, CSS, welcome-email body), email-completeness widget (`preview_text`), tier-aware preflight, domain-connection check, `auto_publish_welcome_email` module flag, four served-page tests carried forward from v1.6.7 scope, reference docs.
**Target release:** v1.7.0 (minor — substantial scaffold redesign + new module variables + new preflight emits + `set-project-field.sh` allow-list changes)
**Dependency:** v1.6.7 shipped (which transitively requires v1.6.6 — `2026-04-26-v1.6.6-upload-multipart.md` and `2026-04-26-v1.6.7-email-rendering-and-form-hide.md`)

## Problem

The v1.6.6 + v1.6.7 surgical fixes make `npm run deploy` actually transfer file bodies (v1.6.6) and existing welcome-email and form resources render correctly (v1.6.7). They do not address four broader incompletenesses revealed by recent deploys:

1. **Scaffold templates are raw static HTML, not HubL.** The framework was designed with `__DOMAIN__/__DM_PATH__` token substitution as a deliberate parameterisation mechanism (added in v1.0.0). Tokens themselves are fine — they're how project-specific values get stamped in. What's missing is HubL primitives: `templateType: page` annotation, `{{ get_asset_url('__DM_PATH__/...') }}` for asset URLs, `{{ standard_header_includes }}` and `{{ standard_footer_includes }}`. Without these, HubSpot serves the templates as static HTML — CSS 404s (URLs constructed manually with `__DOMAIN__/__DM_PATH__` don't resolve to HubSpot's served asset paths), forms degrade silently because the scriptloader runtime never loads, and analytics don't fire. A reference project that uses HubL primitives renders correctly today; the scaffold simply hasn't been updated to that pattern.

2. **Scaffold CSS is single-mode.** Modern landing pages should respect `prefers-color-scheme`. Adding it post-hoc per project is friction; the scaffold should ship with light/dark mode auto-switching as the default, with a `:root` token system that projects override for brand colour choices.

3. **Scaffold welcome-email body is structurally thin.** The current scaffold ships a body shape that passes lint but produces a bare-minimum email — no preheader, no clear CTA, no share/reply prompts, no anatomy at all. Projects then hand-author all of this per-deploy. A good scaffold should encode the anatomy as the default and let projects override copy, not structure.

4. **Preflight is tier-blind.** It demands seven scopes uniformly. On Starter, two further scopes (`marketing-email`, `transactional-email`) needed by `terraform_data.publish_welcome_email` simply don't exist — the API gates them by tier — so demanding them flags as "missing" when the right answer is "skip the auto-publish on Starter." `auto_publish_welcome_email` should default to `true` (preserving v1.6.x Pro+ behaviour) and be downgraded to `false` per-project for Starter consumers via the framework profile. Domain-connection check should also move into preflight: catching `DOMAIN` not present in `/cms/v3/domains` *before* `terraform apply` saves the temp-slug round-trip.

## Design

### Fix 1 — Scaffold templates rewritten as HubL — `scaffold/src/templates/*.html`

Replace current scaffold templates with HubL-correct versions. **Token substitution stays** — the `__DOMAIN__`, `__DM_PATH__`, `__PORTAL_ID__`, `__CAPTURE_FORM_ID__`, etc. tokens still get substituted at build time. The change is what they substitute *into*: HubL function calls, not raw URLs.

Required elements in every scaffold template:

- **First line:** `{# templateType: page #}` annotation. Without this, HubSpot serves the file as static HTML — no HubL compilation.
- **In `<head>`:** `<meta name="color-scheme" content="light dark">` (informs browser of dual-mode support; HubSpot may strip this in serving but it's the correct source-of-truth).
- **In `<head>`, before any custom CSS:** `{{ standard_header_includes }}`. This loads HubSpot's runtime: scriptloader, jQuery (legacy), CSP-safe analytics, etc. Forms depend on it.
- **CSS / JS asset URLs:** `{{ get_asset_url('__DM_PATH__/css/main.css') }}` — never raw `https://__DOMAIN__/__DM_PATH__/css/main.css`. The HubL function resolves to HubSpot's compiled `hub_generated/template_assets/...` URL when the page is served; build-time token substitution still expands `__DM_PATH__` to the project's DM upload path because the substitution regex doesn't match HubL syntax (e.g. matches `__DM_PATH__` inside `{{ get_asset_url('__DM_PATH__/...') }}` and produces `{{ get_asset_url('/myproject/...') }}` after build).
- **Before `</body>`:** `{{ standard_footer_includes }}`.
- **Form embeds:** retain the `hbspt.forms.create` pattern but ensure the embed script tag is loaded via `standard_header_includes` (it is — the manual `<script src="//js-eu1.hsforms.net/...">` tag in current scaffold is redundant and can be removed).
- **Two `hbspt.forms.create` instances by default:** hero (`target: "#signup-form"`) and bottom (`target: "#signup-form-bottom"`). Same form ID. HubSpot dedupes contacts by email on submission, so duplicate hero+bottom submissions resolve to one contact. Conditional on `var.include_bottom_cta` (see Fix 7).

A reference project (existing internal project that uses these primitives correctly) renders today: CSS 200, scriptloader fires, forms initialise. The implementer compares the rewritten scaffold against that reference; the scaffold's templates and the reference project's templates should be structurally equivalent (modulo project-specific copy).

### Fix 2 — Scaffold CSS rewrite — `scaffold/src/css/main.css`

Two structural changes:

**1. Token system via `:root`.** Define design tokens (background, surface, text, border, accent, etc.) as `--*` custom properties on `:root`. All component rules reference tokens, never raw colours.

**2. `@media (prefers-color-scheme: dark)` overrides.** A single block flips the token values. Because component rules reference tokens, the entire stylesheet adapts automatically.

**Default values use placeholder tokens for the brand-specific elements**, not literal colour values — this prevents the scaffold's defaults becoming the de-facto brand for projects that don't override them:

```css
:root {
  --bg: #FFFFFF;
  --surface: #FFFFFF;
  --text: #0A0A0B;
  --border: #0A0A0B;
  --accent: __PRIMARY_ACCENT__;       /* skill substitutes per-project at scaffold time */
  --accent-on: __PRIMARY_ACCENT_ON__; /* contrasting text colour on accent backgrounds */
  --muted: #5A5A5C;
}

@media (prefers-color-scheme: dark) {
  :root {
    --bg: #0A0A0B;
    --surface: #141416;
    --text: #F2F2F0;
    --border: #F2F2F0;
    --accent: __PRIMARY_ACCENT_DARK__;       /* skill substitutes; usually slightly brighter than light-mode accent for contrast */
    --accent-on: __PRIMARY_ACCENT_ON_DARK__;
    --muted: #A0A0A2;
  }
}
```

`__PRIMARY_ACCENT__` and friends are tokens the **skill** substitutes at scaffold time per the brand-direction prompt (skill plan §Step 7). The framework's build pipeline does not substitute these — they're not in `project.config.sh`'s scope. Document this clearly in the scaffold's CSS file header so future contributors don't add them to `build.sh`'s sed pattern.

**`project_source` field hide — defence-in-depth.** Even with the form-level `hidden: true` flag from v1.6.7, scaffold CSS retains broad selectors covering HubSpot Forms v3 markup variants:

```css
[data-field-name="project_source"],
input[name="project_source"],
label[for*="project_source"],
.hs-form-field:has(input[name="project_source"]) {
  display: none !important;
  visibility: hidden;
  position: absolute;
  left: -9999px;
}
```

The form-level flag is primary; the CSS is belt-and-braces for any portal where the flag's render behaviour drifts.

### Fix 3 — Scaffold welcome-email rewrite — `scaffold/src/emails/welcome-body.html`

Replace with the **anatomy template** specified by `references/email-anatomy.md` (new — see Fix 8 below). Inner HTML only — no `<!DOCTYPE>`, `<html>`, `<head>`, or `<body>` wrapper (HubSpot's DnD `Plain_email.html` template provides those plus the legal unsubscribe + physical-address footer).

Anatomy elements (each generated as a labelled section the consumer/skill can swap content into):

1. **Preheader** (hidden span, ~85–110 chars).
2. **View-in-browser link** (small, muted, top): `{{ view_as_page_url }}`.
3. **Brand wordmark / logo** (project's `logo.svg` or text mark — scaffold ships a placeholder).
4. **Headline** — single line confirming user action.
5. **Subhead** — one sentence restating what was signed up for.
6. **Body** — 2–4 short paragraphs.
7. **Primary CTA** — single button. Profile-aware destination (skill substitutes per-profile; scaffold ships a placeholder URL token `__PRIMARY_CTA_URL__`).
8. **Secondary engagement** — share-with-a-friend row (LinkedIn, WhatsApp, forward-via-email). Conditional on a `share_channels` brief field; scaffold ships the row commented out.
9. **Reply prompt** — "Hit reply if X" line.
10. **Sign-off** — team / project name.
11. **Cadence note** — one line setting expectation for future contact frequency.

**No manual UTM parameters in in-email links.** HubSpot auto-tags marketing emails with its own tracking parameters and overwrites manual ones; scaffold should not add `utm_source=welcome_email&...`. UTMs only on share-out links (LinkedIn / WhatsApp / forward), where users land outside HubSpot's email-tracking scope. Document this constraint in `references/email-anatomy.md`.

### Fix 5 — Tier-aware preflight — `scripts/preflight.sh` + `scripts/lib/tier-classify.sh`

Tier classification is non-trivial enough to warrant its own helper. New file:

**`scripts/lib/tier-classify.sh`** — sourced by `preflight.sh`:

```bash
# tier-classify.sh — classify a HubSpot portal's tier from /account-info/v3/details
# Sourced by preflight.sh; emits HUBSPOT_TIER and HUBSPOT_TIER_LABEL via stdout.
#
# Probe values per Prerequisite A; update if observed values differ.
# Output values: starter | pro | ent | ent+tx
# Caller pattern:
#   source "$FRAMEWORK_HOME/scripts/lib/tier-classify.sh"
#   tier="$(classify_tier_from_account_details "$ACCOUNT_INFO_JSON")"

classify_tier_from_account_details() {
  local json="$1"
  local account_type
  account_type=$(echo "$json" | jq -r '.accountType // "UNKNOWN"')
  # ... mapping logic ...
}
```

Three new emits from `preflight.sh`:

**1. `PREFLIGHT_TIER`.** One-shot probe of `/account-info/v3/details`; classify via `tier-classify.sh`. Emit `PREFLIGHT_TIER=<starter|pro|ent|ent+tx>`.

**2. Tier-aware `PREFLIGHT_SCOPES`.** Required scope set varies by tier (verify counts via Prerequisite B; the structure below is the assumption to be confirmed):

- Starter → 7 scopes (current framework requirement)
- Pro → 8 (`+ marketing-email`)
- Ent → 8 (`+ marketing-email` — same set as Pro)
- Ent+TX add-on → 9 (`+ transactional-email`)

On Starter, missing `marketing-email` / `transactional-email` is **expected**, not a failure. Emit `PREFLIGHT_SCOPES=ok-starter` rather than `missing` for that case. The skill reads this and coaches manual UI publish.

**3. `PREFLIGHT_DOMAIN_CONNECTED`.** When project's hosting mode is `custom-domain-primary` (per `HOSTING_MODE_HINT` in `<project>.skillstate.sh`), probe `/cms/v3/domains` for the project's `DOMAIN`. Emit:
- `ok` — domain present in portal with `isUsedForLandingPage: true`
- `missing` — not in portal at all (Starter limit may be the cause; skill suggests system-domain mode)
- `not-primary` — present but not primary for landing pages
- `skipped` — hosting mode is system-domain or iframe; check is N/A

Catches the temp-slug failure mode *before* `terraform apply`, not after.

### Fix 6 — `auto_publish_welcome_email` module flag — `terraform/modules/landing-page/`

**`variables.tf`:**

```hcl
variable "auto_publish_welcome_email" {
  description = <<EOT
Whether to automatically publish the welcome email post-create via the
publish_welcome_email terraform_data resource.

Default true preserves v1.6.5+ behaviour for Pro+ portals where the
publish endpoint requires marketing-email scope. The skill flips this
to false (via set-project-field.sh AUTO_PUBLISH_WELCOME_EMAIL=false)
when preflight detects a Starter tier; on Starter the publish API
endpoint is unavailable (scope is gated by tier), so the email goes
to UI-publish manually after deploy.
EOT
  type    = bool
  default = true
}
```

**`emails.tf`:** Wrap `terraform_data.publish_welcome_email`:

```hcl
resource "terraform_data" "publish_welcome_email" {
  count            = var.auto_publish_welcome_email ? 1 : 0
  triggers_replace = [restapi_object.welcome_email.id]

  provisioner "local-exec" {
    command = <<-EOT
      bash "$HS_LANDER_PROJECT_DIR/scripts/hs-curl.sh" POST "/marketing/v3/emails/${restapi_object.welcome_email.id}/publish"
    EOT
  }
}
```

**Default `true`** preserves v1.6.5–v1.6.7 behaviour for existing Pro+ consumers — they don't lose automation on upgrade. Skill flips to `false` per-project when `PREFLIGHT_TIER=starter` (see Fix 7 — allow-list extension).

### Fix 8 — `include_bottom_cta` module flag

Add a second `hbspt.forms.create` instance near the foot of the scaffold landing template, between FAQ and footer sections (or wherever the natural body-end is — scaffold layout is the project's call). Same form ID, different `target` element ID (`#signup-form-bottom`). HubSpot dedupes contact submissions by email.

`variables.tf` adds:

```hcl
variable "include_bottom_cta" {
  description = "Render a second sign-up form embed near the bottom of the landing page. HubSpot dedupes contacts on submission across both embeds."
  type        = bool
  default     = true
}
```

The variable doesn't directly affect Terraform resource shape (the form is the same form), but it gates whether the scaffold template renders the second embed. In `scaffold/src/templates/landing-page.html`, wrap the bottom-CTA section in a Jinja-style conditional that `build.sh` resolves: but since `build.sh` is sed-based and doesn't run a templating engine, the simpler approach is to ship the bottom CTA always-on in the scaffold and document that projects opt out by deleting the section. Treat `include_bottom_cta` as advisory metadata in the project profile (consumed by skill for coaching) rather than wiring it into Terraform — a future v1.8.0 plan can introduce a real templating layer if needed. Document this scoping decision in the scaffold CSS header.

**`references/hubspot-api-quirks.md`** — empirical surface quirks discovered across releases. Implementer checks if a similar file exists in the framework repo (one exists in the TSC-pre-module project per CLAUDE.md context); update if so, otherwise create. Contents:

- POST and PATCH responses on `/marketing/v3/emails` omit `flexAreas` from the response payload even when the data was accepted; a fresh GET shows truth. Trust GET, not the write-response.
- PATCH on `/marketing/v3/emails` strips `flexAreas` to `{}` regardless of payload — the surface accepts widget content edits but rejects layout edits silently. Layout placement is achievable only via POST (create path) or via UI editing.

- Widget ordering in DnD emails is established by position in `flexAreas.main.sections[].columns[].widgets` array, not by a widget-level `order` field.
- `templateType: page` annotation is required for HubL rendering. Templates without it are served as static HTML — CSS 404s, forms degrade, scriptloader doesn't fire.
- `/marketing/v3/forms/{id}` does not expose form follow-up email setting. Form follow-up is UI-only on every tier.
- `marketing-email` and `transactional-email` scopes are gated by HubSpot subscription tier. Cannot be granted on Starter regardless of Service Key permissions.
- `/account-info/v3/details` `accountType` field values per tier (per Prerequisite A).
- HubSpot rendering cache: Design Manager template uploads can take ~30–60 seconds to propagate to served landing pages. After `npm run deploy`, post-deploy verification should wait or accept stale-cache responses for the first probe.

## Migration

Module variables added (`email_preview_text`, `auto_publish_welcome_email`, `include_bottom_cta`) all have defaults that preserve v1.6.7 behaviour for consumers who don't set them. No mandatory migration steps for existing Pro+ projects.

**Starter consumers:** the skill (separate plan) flips `AUTO_PUBLISH_WELCOME_EMAIL=false` in their project profile on next deploy. Without that flip, the `publish_welcome_email` step continues to fail on missing scopes (harmless — leaves email in DRAFT for UI publish, same as v1.6.5–v1.6.7 behaviour). Even if a Starter consumer upgrades to v1.7.0 framework before the skill catches up, nothing regresses.

**Scaffold rewrite:** affects `scaffold-project.sh` output for new projects only. Existing projects' `src/templates/*.html`, `src/css/main.css`, `src/emails/welcome-body.html` are left as-is. Document an optional migration path in CHANGELOG: consumers who want HubL templates in an existing project can copy the new scaffold files in and re-stamp project-specific values.
